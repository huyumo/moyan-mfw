## 三、冲突点分析

### 3.1 命名冲突

#### 🔴 严重：API 层组织方式冲突

**01-编码基础规范.md (第 91-92 行)：**
```
├── apis/              # API 层（由 moyan-api 库自动生成）
│   └── index.ts
```

**06-API 调用规范.md (第 59-66 行)：**
```
src/apis/
├── micro-system/           # 按微服务模块划分
│   ├── index.ts            # API 类定义
│   └── schemas.ts          # 类型定义
```

**冲突：**
- 01-文档说 API 层「由 moyan-api 自动生成」，只需要 `index.ts`
- 06-文档要求手动编写 API 类，需要 `index.ts` + `schemas.ts`

**影响：** 开发者不清楚 API 层是自动生成还是手动编写。

**建议：**
1. 明确区分「moyan-api 生成的 API」和「手动编写的 API」
2. 或者统一为一种模式

#### ⚠️ 中等：目录结构描述不一致

**01-编码基础规范.md (第 116 行)：**
```
├── views/             # 内置页面视图（每个页面是一个目录）
```

**01-编码基础规范.md (第 162 行)：**
```
frontend/src/
├── views/             # 业务视图（每个页面是一个目录）
```

**问题：** 「内置页面」和「业务视图」的边界不清晰，AI 可能混淆框架层和业务层。

### 3.2 概念冲突

#### 🔴 严重：路由生成模式描述不一致

**05-路由配置指南.md (第 7 行)：**
> 本框架采用**约定优于配置**的原则，通过自动扫描页面配置文件生成路由和菜单

**01-编码基础规范.md (第 215-338 行)：**
> 模块路由配置（可选）：在模块目录定义 `index.ts` 导出模块配置

**问题：**
- 05-文档说「自动扫描生成，无需手动配置」
- 01-文档说「模块配置（可选）」

**影响：** AI 可能错误地建议用户手动配置路由表。

**建议：** 明确「自动扫描」是默认模式，「手动配置」是特殊场景。

### 3.3 代码示例冲突

#### ⚠️ 中等：Composable 导入方式

**03-Composables 规范.md (第 99 行)：**
```typescript
import { useUserList } from './composables/useUserList';
```

**06-API 调用规范.md (第 227 行)：**
```typescript
import { ApiUserList, ApiUserInfoFindOne, type User } from '@/apis/micro-system'
```

**问题：**
- 03-文档使用相对路径 `./composables/`
- 06-文档使用别名路径 `@/apis/`

**建议：** 统一导入路径规范，或说明何时使用相对路径/别名路径。
