---
name: 用户工作模式与质量要求
description: 用户明确表达的工作方式偏好和质量标准
type: feedback
---

**规则 1：停止无效反思，只执行有效行动**

不要过多解释、反思或道歉，直接执行任务。

**规则 2：主动识别任务，不要询问**

- 不要问"你想让我做什么"
- 不要问"接下来做什么"
- 主动识别当前需要完成的任务并执行
- 用户说："我不知道呀！这是你的工作~，你问我，我问谁呢 ？"

**规则 3：严格对齐文档**

- 必须严格按照需求文档开发
- 不得擅自假设或跳过文档要求
- 不得"瞎写"或忽悠过关
- 用户说："严格对齐文档，不要瞎写，我发现我不支持具体问题，你们就忽悠过关吗？"

**规则 4：高质量标准**

- 功能完成度 50%
- 代码质量 30%
- 用户对这个质量打分，说明期望很高

**规则 5：立即执行整改**

当发现接口不对齐时：
- 接口先对齐文档
- 前端对齐文档
- 前后端对齐
- 开发与测试对齐
- 立即开始，不拖延

**规则 6：用户打断/指出错误 = 必须保存**

当用户中途打断、指出错误时，这些反馈必须保存到记忆，不需要用户明确说"请记住"。

**规则 7：apis 目录严禁手动修改（铁律）**

- `apis/` 目录下的文件**必须由 moyan-api 自动生成**
- 严禁手动修改 apis 目录下的任何文件
- 如果 API 类型有缺失或错误：
  1. 通知后端开发人员调整 Swagger
  2. 后端运行 `moyan api:generate` 生成 Swagger
  3. 前端调用 moyan-api 重新生成 apis
  4. 替换 `apis/sys/` 目录下的自动生成代码
- 这是铁律，必须严格执行
- 违反此规则会导致前后端类型不一致

**规则 8：查看相关文档**

当用户问"是否记得"而我不记得时：
- 立即查看相关文档
- 文档位置：`docs/03-框架规范/01-前端规范/06-API 调用规范.md`
- 文档位置：`docs/05-问题追踪/01-API类型定义问题.md`
- 不要等待用户提醒

**工作流程（apis 变更）**

```
后端修改 Swagger/OpenAPI 注解
       ↓
后端提交代码
       ↓
后端运行 moyan api:generate
       ↓
后端通知前端
       ↓
前端调用 moyan-api 重新生成
       ↓
前端替换 apis/sys/ 目录
       ↓
测试验证
```

**如何应用**：
- 每次会话开始时读取记忆
- 严格遵守上述规则
- 主动识别需要保存的反馈并保存
- apis 目录手动修改后立即回滚
