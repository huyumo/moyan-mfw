## 目的

规范任务文件的格式和更新流程，确保任务状态清晰可追踪。

---

## TASK.md 格式规范

### 文件头（YAML Front Matter）

```yaml
---
task: 任务名称（必填）
status: pending | in_progress | blocked | completed（必填）
priority: P0 | P1 | P2 | P3（必填）
started: YYYY-MM-DD（必填）
updated: YYYY-MM-DD HH:mm（必填）
session: 会话 ID（必填）
lock: 时间戳（可选，用于多会话锁）
assignee: @角色（可选）
---
```

### 字段说明

| 字段 | 说明 | 可选值 |
|------|------|--------|
| task | 任务名称，一句话描述 | 任意字符串 |
| status | 当前状态 | pending, in_progress, blocked, completed |
| priority | 优先级 | P0, P1, P2, P3 |
| started | 任务开始日期 | YYYY-MM-DD |
| updated | 最后更新时间 | YYYY-MM-DD HH:mm |
| session | 最后更新的会话 ID | 会话 ID 字符串 |
| lock | 会话锁时间戳 | Unix 时间戳（秒） |
| assignee | 负责人 | @角色 |

## 必选章节

### 1. 当前目标

**说明**：一句话描述当前任务的核心目标

**格式**：
```markdown
## 当前目标
[一句话描述]
```

**示例**：
```markdown
## 当前目标
开发墨焱框架前端组件库，参考网约车 admin 项目组件，建立规范化组件体系。
```

---

### 2. 已完成

**说明**：已完成的任务项列表

**格式**：
```markdown
## 已完成
- [x] 任务 1
- [x] 任务 2
```

---

### 3. 进行中

**说明**：正在进行中的任务，每个任务包含状态、相关文件、阻塞点

**格式**：
```markdown
## 进行中
- [ ] 任务名称（优先级）
  - 相关文件：`路径`
  - 当前状态：描述
  - 阻塞点：无/具体阻塞
```

---

### 4. 待开始

**说明**：待开始的任务列表，标注优先级

**格式**：
```markdown
## 待开始
- [ ] 任务名称（P0/P1/P2/P3）
```

---

### 5. 相关文件

**说明**：与当前任务相关的文件路径列表

**格式**：
```markdown
## 相关文件
- `路径 1`
- `路径 2`
```

---

## 可选章节

### 关键决策

**说明**：任务执行过程中的重要决策记录

**格式**：
```markdown
## 关键决策
- 决策内容 + 原因
```

---

### 备注

**说明**：其他需要说明的信息

**格式**：
```markdown
## 备注
[其他信息]
```

## 任务状态定义

| 状态 | 说明 | 流转条件 |
|------|------|----------|
| pending | 待开始 | 任务已创建，等待执行 |
| in_progress | 进行中 | 任务正在执行 |
| blocked | 阻塞 | 遇到阻塞，等待解决 |
| completed | 已完成 | 任务完成并验收 |

**状态流转图**：
```
pending → in_progress → blocked → in_progress → completed
                     ↘--------------↗
```

---

## 优先级定义

| 优先级 | 说明 | 响应时间 | 示例 |
|--------|------|----------|------|
| P0 | 紧急且重要 | 立即处理 | 线上 Bug、核心功能 |
| P1 | 重要不紧急 | 24 小时内 | 关键功能、重要文档 |
| P2 | 紧急不重要 | 48 小时内 | 一般优化、文档更新 |
| P3 | 不紧急不重要 | 待排期 | 技术债务、体验优化 |

## 更新规则

### 会话开始
- 读取 TASK.md
- 向用户确认任务状态
- 如状态与实际不符，立即更新

### 会话中
- 任务完成时立即标记 `[x]`
- 遇到阻塞时更新 `blocked` 状态
- 重大决策时记录到"关键决策"

### 会话结束
- 必须更新 `updated` 和 `session` 字段
- 必须更新进度（已完成/进行中/待开始）
- 必须记录下一步行动

### 任务动态调整（新增）

**1. 添加任务**

| 场景 | 操作 | 示例 |
|------|------|------|
| 添加待开始任务 | 添加到"待开始"列表，记录添加时间和原因 | `- [ ] 任务名（P1）← 2026-03-30 添加` |
| 插入紧急任务 | 当前任务标记"暂停"，新任务添加到"进行中"，标记"插队" | `🔥 插队任务 XXX（P0）` |
| 添加关联子任务 | 在父任务下增加子任务项 | `- [ ] 子任务 1（父任务：XXX）` |

**2. 删除任务**

| 场景 | 操作 | 示例 |
|------|------|------|
| 删除待开始任务 | 从列表中移除，在"备注"中记录删除原因 | `备注：任务 XXX 已取消（原因：...）` |
| 删除进行中任务 | 移除任务，记录已完成部分和取消原因 | `备注：任务 XXX 中止，已完成子任务 YYY` |
| 删除已完成任务 | 不允许删除，保留历史记录 | - |

**3. 任务状态调整**

| 场景 | 操作 | 示例 |
|------|------|------|
| 任务暂停 | 标记状态为"⏸️ 已暂停"，记录暂停原因 | `- [ ] 任务名 - ⏸️ 已暂停（原因：...）` |
| 任务恢复 | 移除暂停标记，恢复正常状态 | `- [ ] 任务名（P1）` |
| 优先级调整 | 更新优先级标记 | `- [ ] 任务名（P0 ← P2）` |

**4. 变更记录格式**

在 TASK.md 中增加"变更记录"章节（可选）：

```markdown
## 变更记录
| 时间 | 变更类型 | 变更内容 | 原因 |
|------|----------|----------|------|
| 2026-03-30 22:45 | 添加任务 | 新增任务 XXX | 用户临时要求 |
| 2026-03-30 23:00 | 删除任务 | 移除任务 YYY | 用户需求变更 |
| 2026-03-30 23:15 | 暂停任务 | 任务 ZZZ 暂停 | 等待外部依赖 |
```

---

## 违规处理

| 违规项 | 处理动作 |
|--------|----------|
| TASK.md 不存在 | 停止工作，先创建 |
| 文件格式错误 | 停止工作，先修复 |
| 未更新 TASK.md | 会话结束前强制提醒 |
| 状态与实际不符 | 立即更新 |

## 任务文件位置

### 活跃任务（双层设计）

**第一层：`TASK.md`（根目录）**

- **用途**：记录当前活跃任务状态，供会话开始快速读取
- **特点**：轻量级，快速更新，只关注当前任务
- **文件结构**：
  - YAML Front Matter（必填字段）
  - 当前目标（一句话描述）
  - 已完成（仅保留本期任务摘要 + 历史归档引用）
  - 进行中（当前正在执行的任务）
  - 待开始（等待执行的任务）
  - 相关文件（含历史归档引用）

**第二层：`docs/04-项目实施/05-任务追踪/`**

- **用途**：任务详情和历史归档
- **特点**：重量级，完整记录，可长期追溯
- **文件类型**：
  1. **任务详情文档**：`YYYY-MM-DD-HHMM-任务名称.md`（每个任务一个）
  2. **TASK 快照归档**：`archived/TASK-YYYY-MM-DD-HHMM.md`（任务周期结束时）

---

### 任务详情文档

- 位置：`[project]/docs/04-项目实施/05-任务追踪/YYYY-MM-DD-HHMM-任务名称.md`
- 特点：重量级，完整记录
- 用途：任务详情、决策记录、历史追溯

---

### TASK.md 历史归档

- 位置：`[project]/docs/04-项目实施/05-任务追踪/archived/TASK-YYYY-MM-DD-HHMM.md`
- 触发条件：当 `TASK.md` 中所有任务均已完成时
- 用途：保存 TASK.md 完整快照，便于追溯

## TASK.md 归档与重置流程

### 归档触发条件

当满足以下条件时，执行 TASK.md 归档与重置：

1. 用户明确要求归档任务周期
2. 当前主要任务已完成或进入新阶段
3. 用户确认可以开始新的任务周期

**注意**：归档时可能存在未完成的遗留任务，这些任务需要保留到新周期。

---

### 归档步骤

```
1. 检查是否有遗留任务
   → 遍历"进行中"和"待开始"列表，识别未完成任务
   → 将遗留任务单独记录（含优先级、状态、原因）
       ↓
2. 创建 TASK.md 完整快照
   → 复制到 docs/04-项目实施/05-任务追踪/archived/TASK-YYYY-MM-DD-HHMM.md
       ↓
3. 重置 TASK.md
   - 清空"已完成"列表，仅保留归档引用
   - 保留遗留任务到"待开始"列表，标注"← 遗留"
   - 更新 YAML Front Matter（新任务等待输入）
       ↓
4. 向用户确认归档完成
   - "TASK.md 已归档至 [归档路径]，遗留 [N] 个任务已保留"
```

---

### 遗留任务处理规范

**定义**：遗留任务是指归档时仍处于"进行中"或"待开始"状态的任务。

**处理规则**：

| 遗留任务类型 | 处理方式 | 示例 |
|--------------|----------|------|
| 低优先级任务（P3） | 保留到新周期"待开始"列表 | `- [ ] CI/CD 集成测试（P3）← 遗留` |
| 阻塞任务 | 保留到新周期，记录阻塞原因 | `- [ ] 任务 XXX - ⏸️ 阻塞（原因：...）← 遗留` |
| 暂停任务 | 保留到新周期，保持暂停状态 | `- [ ] 任务 XXX - ⏸️ 已暂停 ← 遗留` |
| 高优先级未完成任务 | 提醒用户确认是否保留或重新规划 | 询问用户后决定 |

**遗留任务标注格式**：

```markdown
## 待开始
### 上期遗留任务（P3）
- [ ] 【P3】任务名称 ← 遗留
  - 遗留原因：低优先级，待排期

### 本期新增任务
- [ ] 【P1】新任务名称
```
