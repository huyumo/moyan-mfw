## 目的

规范项目记忆的保存流程和格式，确保跨会话知识传承。

---

## 记忆类型

### 1. user - 用户记忆

**内容**：用户角色、偏好、知识、技能

**示例**：
```markdown
---
name: 用户角色与技能
description: 用户的技术背景和角色定位
type: user
---

用户是资深前端工程师，有 10 年 Go 开发经验，首次接触 React/Vue 前端框架。

**技术栈**：
- 精通：Go、TypeScript
- 学习中：Vue 3、React
- 目标：构建统一的前端组件库

**工作风格**：
- 偏好详细的技术文档
- 重视类型定义和代码规范
- 喜欢先设计后实现
```

**保存时机**：
- 了解用户角色时
- 学习用户偏好时
- 发现用户技能背景时

---

### 2. feedback - 反馈记忆

**内容**：用户给出的工作反馈、纠正、确认

**示例**：
```markdown
---
name: 会话流程强制规则
description: 会话开始/结束必须遵循的流程
type: feedback
---

**规则**：会话开始必须读取 TASK.md 并确认任务，会话结束必须更新 TASK.md。

**原因**：用户希望确保跨会话连续性，避免每次重新开始。

**如何应用**：每次会话开始时，自动读取 TASK.md 并向用户确认状态。
```

**保存时机**：
- 用户纠正工作方法时
- 用户确认工作方式时
- 用户表达偏好时

---

### 3. project - 项目记忆

**内容**：项目目标、决策、约束条件、进展状态

**示例**：
```markdown
---
name: 前端组件库技术决策
description: 组件库技术栈选择和技术方案
type: project
---

**技术栈选择**：
- 基础组件：TSX + defineComponent
- 业务组件：<script setup lang="ts">
- Vue 3.4+、TypeScript 5.3+

**原因**：
- TSX 提供更好的类型推断
- defineComponent 支持完整组件选项
- 符合项目统一规范

**如何应用**：开发新组件时，严格遵循技术栈选择。
```

**保存时机**：
- 项目重要决策完成时
- 技术方案确定时
- 项目目标变更时

---

### 4. reference - 参考记忆

**内容**：外部资源位置、工具链接、文档地址

**示例**：
```markdown
---
name: 参考组件源码位置
description: 网约车 admin 项目组件源码路径
type: reference
---

**源码位置**：`E:\YunYi\网约车自选小程序\代码\admin\src\components\`

**核心组件**：
- FormCard - 动态表单生成器
- PageSceneV2 - 页面场景模板
- TableList - 动态表格列表
- Popup - 命令式弹窗

**如何应用**：开发新组件时，参考其设计思路和实现方式。
```

**保存时机**：
- 发现外部资源时
- 配置工具链接时
- 记录文档地址时

---

## 记忆文件格式

### 文件头（YAML Front Matter）

```yaml
---
name: 记忆名称（必填）
description: 一句话描述（必填）
type: user | feedback | project | reference（必填）
---
```

### 文件内容结构

**feedback/project 类型**：
```markdown
**规则/事实**：[具体内容]

**原因**：[为什么这样规定/决策]

**如何应用**：[具体应用场景和方式]
```

**user 类型**：
```markdown
[用户相关信息和背景]

**技能/偏好**：
- 列表形式

**工作风格**：
- 列表形式
```

**reference 类型**：
```markdown
**位置/链接**：[资源地址]

**内容说明**：[资源内容描述]

**如何应用**：[使用场景]
```

## 记忆保存流程

### 步骤 1：识别保存内容

**触发条件**：
- 用户说"请记住..."
- 发现用户偏好
- 项目重要决策
- 新的工作模式约定

### 步骤 2：确定记忆类型

根据内容选择：user / feedback / project / reference

### 步骤 3：创建记忆文件

**位置**：
- 用户级：`C:\Users\16898\.claude\memory\`
- 项目级：`[project]/.claude/memory/`

**动作**：
1. 检查是否有重复记忆
2. 创建新文件
3. 写入内容

### 步骤 4：更新 MEMORY.md

**位置**：`[记忆目录]/MEMORY.md`

**格式**：
```markdown
- [记忆名称](文件名.md) — 一句话描述
```

### 步骤 5：询问用户确认

**话术**：
> "我已将 [内容] 保存到记忆 [文件名.md]，是否需要修改？"

## 记忆读取规则

### 会话开始时

**动作**：读取 `MEMORY.md`

**用途**：
- 了解用户偏好
- 回忆项目决策
- 应用工作规则

### 遇到不确定时

**动作**：查阅相关记忆文件

**示例**：
- 不确定用户偏好 → 读取 user 类型记忆
- 不确定技术方案 → 读取 project 类型记忆
- 不确定工作规则 → 读取 feedback 类型记忆

---

## 记忆更新规则

### 何时更新

- 用户明确要求修改
- 记忆内容与实际不符
- 项目决策变更

### 如何更新

1. 读取原记忆文件
2. 修改内容
3. 更新 `name` 和 `description`（如有变化）
4. 更新 MEMORY.md 中的描述（如有变化）

---

## 记忆删除规则

### 何时删除

- 用户要求删除
- 记忆内容过时
- 记忆内容错误

### 如何删除

1. 删除记忆文件
2. 从 MEMORY.md 中移除对应行

## 跨设备移植

本项目所有规则文件均使用**相对路径**，存放在 `.claude/rules/` 目录下。

**移植方式**：
1. 复制整个 `.claude/` 目录到新设备
2. 或将项目目录整体拷贝（包含 Git 历史）
3. 规则文件自动生效，无需修改路径

**用户级记忆**（可选）：
- 如需要跨项目共享记忆，可在用户级记忆目录创建符号链接
- 或手动复制 `memory/` 目录到新设备
