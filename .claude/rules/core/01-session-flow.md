## 目的

确保每个会话开始时快速了解当前任务状态，会话结束时记录工作成果，实现跨会话工作连续性。

---

## 会话开始时必须执行

### 步骤 1：读取任务文件

**动作**：读取 `TASK.md`

**检查项**：
- [ ] 文件是否存在
- [ ] 文件格式是否正确（YAML Front Matter）
- [ ] 当前任务名称和状态
- [ ] 会话锁（如 5 分钟内有其他会话写入，警告用户）

**输出**：
> "我看到当前任务是 [任务名称]，状态是 [状态]，优先级是 [优先级]"

---

### 步骤 2：确认任务状态

**动作**：向用户确认

**话术**：
> "TASK.md 中的任务状态是否准确？如有变化请告诉我，我先更新再开始工作。"

**分支处理**：
- 如用户说"任务有变化" → 更新 TASK.md 后再继续
- 如用户说"正确" → 继续步骤 3
- 如用户说"没有 TASK.md" → 创建 TASK.md

---

### 步骤 3：确认会话目标

**动作**：询问本次会话计划完成什么

**话术**：
> "今天/这次会话计划完成什么任务？"

**记录**：在 TASK.md 中记录"本次会话目标"

---

### 步骤 4：开始工作

**条件**：以上 3 个步骤全部完成后

**动作**：开始执行用户任务

---

### 步骤 4.5：任务执行前更新（新增）

**当用户确认方案后、开始执行前，必须执行：**

```
1. 确认用户已同意方案
       ↓
2. 立即更新 TASK.md
   - 标记任务状态变更（如：待开始 → 进行中）
   - 记录当前子任务/步骤
   - 更新 updated 和 lock 字段
       ↓
3. 向用户确认已更新
   - "已更新 TASK.md，任务状态变更为 [X]，现在开始执行"
       ↓
4. 开始执行任务
```

**违规处理**：用户应提醒"请先更新 TASK.md 再执行"

**说明**：一次会话中可能执行多个任务，每个任务开始前都应更新 TASK.md。

---

### 步骤 5：自我反思（任务完成后）

**条件**：任务执行完成后、更新 TASK.md 前

**动作**：反向推理，从他人角度验证质量和逻辑

**检查清单**：
- [ ] 代码/文档是否符合规范
- [ ] 是否遗漏边界情况
- [ ] 是否有潜在问题
- [ ] 验收标准是否全部满足

**输出**：
- 如发现问题 → 立即修复
- 如无问题 → 继续步骤 5.5

---

### 步骤 5.5：强制自测试验证（新增，权重最高）

**条件**：自我反思通过后、任务归档前

**强制执行**：此步骤为**强制门禁**，未完成不得进入任务归档

**必须执行的检查清单**：

| 序号 | 检查项 | 执行命令/方法 | 通过标准 |
|------|--------|--------------|---------|
| 1 | **单元测试** | `pnpm vitest run` 或 `pnpm test:unit` | 所有测试通过，无失败 |
| 2 | **类型检查** | `pnpm typecheck` 或 `pnpm typecheck:vue` | 无类型错误 |
| 3 | **项目启动验证** | `pnpm dev`（前端）+ `pnpm start:dev`（后端） | 服务正常启动，页面可访问 |
| 4 | **功能冒烟测试** | 手动验证核心功能 | 主要功能流程正常 |

**验证流程**：
```
1. 运行单元测试
   ↓ 通过？
   ↓ 否 → 修复测试/代码 → 重新执行步骤 5.5
   ↓ 是
2. 运行类型检查
   ↓ 通过？
   ↓ 否 → 修复类型错误 → 重新执行步骤 5.5
   ↓ 是
3. 启动项目验证
   ↓ 正常？
   ↓ 否 → 修复启动问题 → 重新执行步骤 5.5
   ↓ 是
4. 功能冒烟测试
   ↓ 正常？
   ↓ 否 → 修复功能问题 → 重新执行步骤 5.5
   ↓ 是
5. 自测试通过，进入步骤 6 任务归档
```

**强制门禁规则**：
- ⛔ **禁止跳过**：任何理由不得跳过此步骤
- ⛔ **禁止虚假报告**：必须真实执行测试，不得伪造结果
- ⛔ **禁止绕过**：测试失败时必须修复，不得强行归档

**违规处理（最严重级别）**：
| 违规场景 | 处理动作 | 后果 |
|----------|---------|------|
| 跳过自测试直接归档 | 立即停止，退回步骤 5.5 | 任务不计入完成 |
| 未执行测试就报告通过 | 记录违规，重新执行 | 信任度降低 |
| 测试失败后仍强行提交 | 拒绝提交，强制回滚 | 代码不合并 |
| 多次违反此规则 | 用户介入审查 | 可能暂停工作权限 |

**交付话术模板**：
> "开发完成。自测试验证结果：
> - 单元测试：91/91 通过 ✅
> - 类型检查：无错误 ✅
> - 项目启动：后端 http://localhost:3000，前端 http://localhost:5174 ✅
> - 功能验证：核心功能正常 ✅
> 请验收。"

---

### 步骤 6：任务归档（新增）

**当任务完成并通过自我反思后，必须执行：**

```
1. 确认任务已完成（用户验收通过）
       ↓
2. 检查任务详情文档是否存在
   - 文件命名：docs/04-项目实施/05-任务追踪/YYYY-MM-DD-HHMM-任务名称.md
   - 如不存在 → 创建
   - 如存在 → 更新最终状态
       ↓
3. 更新任务详情文档
   - 标记状态：已完成
   - 填写验收记录
   - 记录关键决策和交付物
       ↓
4. 更新 TASK.md
   - 将任务从"进行中"移至"已完成"
   - 记录下一步行动
   - 更新 updated 和 lock 字段
       ↓
5. 提交 Git（如任务详情文档有变更）
       ↓
6. 向用户确认归档完成
   - "任务已归档至 [文件路径]，TASK.md 已更新"
```

**文件命名示例**：
- `2026-03-30-1430-前端组件库开发.md`
- `2026-03-30-1515-API 接口设计.md`

---

## 会话结束时必须执行

### 步骤 1：工作总结

**动作**：列出本次会话完成的所有工作项

**格式**：
```
## 本次会话完成

1. [工作项 1]
2. [工作项 2]
3. [工作项 3]
```

---

### 步骤 2：更新任务文件

**动作**：更新 `TASK.md`

**必填项**：
- `updated` 字段：当前日期时间
- `session` 字段：当前会话 ID
- 已完成列表：标记新完成的任务
- 进行中列表：更新状态
- 下一步行动：记录接下来要做什么

---

### 步骤 3：记忆保存检查

**动作**：检查本次会话是否有需要保存的内容

**检查触发条件**：
- 用户说"请记住..."
- 发现用户偏好
- 项目重要决策
- 新的工作模式约定

**话术**：
> "本次会话有 [X 内容]，是否需要保存到记忆？"

---

## 违规处理

| 违规场景 | 处理动作 | 级别 |
|----------|----------|------|
| **跳过强制自测试验证（步骤 5.5）** | **立即停止，强制退回执行** | **P0 - 致命** |
| **虚假报告自测试结果** | **记录违规，重新执行，用户通报** | **P0 - 致命** |
| **测试失败后强行提交** | **拒绝提交，强制回滚代码** | **P0 - 致命** |
| 未读 TASK.md 就开始工作 | 停止，先读取 TASK.md | P1 - 严重 |
| TASK.md 不存在 | 停止，先创建 TASK.md | P1 - 严重 |
| 未确认任务状态 | 暂停，先确认 | P1 - 严重 |
| 未确自我反查 | 必须检查 | P2 - 一般 |
| 未更新 TASK.md 就想结束 | 提醒用户更新 | P2 - 一般 |
| 未总结工作 | 提醒总结 | P3 - 提示 |

**P0 级违规说明**：
强制自测试验证是代码质量的最后防线，跳过或伪造将直接导致缺陷流入生产环境。任何 P0 级违规必须立即纠正，并记录备案。

## 会话 ID 获取

从以下位置获取当前会话 ID：

1. `[用户目录]/.claude/sessions/[pid].json` 中的 `sessionId`
2. 或 `[用户目录]/.claude/ide/[pid].lock` 中的内容

如无法获取，使用时间戳生成：`session-YYYYMMDD-HHMMSS`

## 多会话锁机制

### 问题
多个 VSCode 窗口 = 多个独立会话 → 可能并发写入 TASK.md

### 解决方案

在 TASK.md 头部记录：
```yaml
session: be2d86c6-3b77-43f3-bfba-6fadbea86dfb
lock: 1711785600  # Unix 时间戳（秒）
```

### 写入前检查
1. 读取当前 `lock` 时间戳
2. 计算 `当前时间 - lock`
3. 如 < 300 秒（5 分钟）→ 警告：
   > "检测到有其他会话正在工作（lock 时间戳显示 5 分钟内有写入），是否继续？"

### 写入时更新
更新 `lock` 为当前时间戳

## 附录：会话流程图

```
会话开始
    ↓
读取 TASK.md
    ↓
检查锁状态 → 有冲突 → 警告用户
    ↓
向用户确认任务状态
    ↓
任务有变化？→ 是 → 更新 TASK.md
    ↓ 否
确认会话目标
    ↓
记录到 TASK.md
    ↓
开始工作

... 工作过程 ...

任务完成
    ↓
自我反思（步骤 5）
    ↓
═══════════════════════════════════════
    【强制自测试验证 - 步骤 5.5】
    单元测试 → 类型检查 → 启动验证 → 功能冒烟
    ⛔ 任何失败必须修复后重新验证
═══════════════════════════════════════
    ↓ 全部通过
任务归档（步骤 6）
    ↓
提交 Git
    ↓
会话结束
```

**关键控制点**：
- 🔒 步骤 5.5 为强制门禁，无绕过机制
- 🔄 测试失败必须回到修复环节，重新验证
- ✅ 全部通过后方可进入任务归档
