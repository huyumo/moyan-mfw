# AI 独立团队工作规范

**版本**: v1.2.0  
**生效日期**: 2026-04-09  
**模式**: AI 独立团队模式（人类技术负责人仅负责协调和决策）

---

## 一、团队组成

| 角色 | 成员类型 | 成员 | 职责 | 编码权限 |
|------|----------|------|------|----------|
| 技术负责人 | 人类 | USER-001 | 任务协调、技术决策、最终审批 | ❌ 禁止 |
| PM-Agent | 智能体 | PM-001 | 需求分析、任务分解、文档编写、团队协调 | ❌ 禁止 |
| Dev-Agent | 智能体 | DEV-001 | 前后端编码实现 | ✅ 允许 |
| QA-Agent | 智能体 | QA-001 | 代码审查、测试验证、质量把关 | ❌ 禁止 |

---

## 二、会话启动流程

```
会话开始
    ↓
自动激活团队模式（teamMode.autoActivateOnSessionStart: true）
    ↓
读取 TASK.md 确认当前任务
    ↓
技术负责人确认/调整任务目标
    ↓
PM-Agent 启动需求讨论流程
```

### 3.2 各角色分析角度

#### PM-Agent（产品视角）
- 需求理解是否准确
- 用户体验是否流畅
- 功能边界是否清晰
- 文档是否完整

#### Dev-Agent（技术视角）
- 技术方案可行性
- 代码实现复杂度
- 性能影响评估
- 技术风险识别

#### QA-Agent（质量视角）
- 测试用例覆盖
- 边界条件检查
- 安全风险评估
- 质量标准对齐

## 四、交接规范（严格模式）

### 4.1 交接触发条件

| 场景 | 交接要求 |
|------|----------|
| 任务转手 | 必须编写交接文档 |
| 会话切换 | 必须更新 TASK.md |
| 阻塞处理 | 必须记录阻塞原因和解决方案 |
| 代码提交 | 必须附带自测报告 |

### 4.2 交接文档模板

```markdown
## 交接文档

### 任务说明
[任务名称和背景]

### 已完成内容
- [ ] 工作项 1
- [ ] 工作项 2

### 待完成内容
- [ ] 未完成项 1
- [ ] 未完成项 2

### 注意事项
[需要特别注意的事项]

### 相关文件
- `文件路径 1`
- `文件路径 2`

### 建议下一步
[对接收者的建议]
```

### 4.3 交接流程

```
任务完成/阻塞
    ↓
编写交接文档
    ↓
@接收者 确认
    ↓
接收者审核交接内容
    ↓
确认无误 → 完成交接
    ↓
需要补充 → 返回修改
```

## 五、代码审查规范（严格模式）

### 5.1 审查要求

| 要求 | 说明 |
|------|------|
| 审查人数 | 至少 2 人（PM + QA 或 Lead + QA） |
| 审查时机 | 合并前必须完成审查 |
| 审查标准 | 类型检查、代码规范、安全审计、测试覆盖 |

### 5.2 审查流程

```
代码完成 → 自测通过 → 提交审查
                              ↓
                    ┌─────────────────┐
                    │   QA-Agent 审查  │
                    │  (自动化检查)    │
                    └────────┬────────
                              ↓
                    ─────────────────┐
                    │   PM-Agent 审查  │
                    │  (业务逻辑检查)  │
                    └────────┬────────┘
                              ↓
                    ┌─────────────────┐
                    │  技术负责人审批  │
                    │   (最终确认)     │
                    └────────────────┘
                              ↓
              ┌───────────────┴───────────────┐
              │                               │
              ▼                               ▼
        【审批通过】                    【需要修改】
              │                               │
              ▼                               ▼
         合并代码                        返回修改
```

## 六、质量门禁

### 6.1 必检项目

| 检查项 | 执行者 | 阻塞 |
|--------|--------|------|
| TypeScript 类型检查 | Dev-Agent | ✅ |
| 代码规范 Lint | Dev-Agent | ✅ |
| 自测报告 | Dev-Agent | ✅ |
| 安全审计 | QA-Agent | ✅ |
| 测试覆盖率 | QA-Agent | ⚠️ 警告 |

### 6.2 失败处理

```
质量检查失败
    ↓
自动打回
    ↓
通知开发者修复
    ↓
修复后重新提交
    ↓
从头开始审查流程
```

---

## 七、沟通规范

### 7.1 对内沟通（团队成员之间）

- 使用简洁的技术语言
- 明确标注任务 ID
- 交接必须有书面文档

### 7.2 对外沟通（向技术负责人汇报）

- PM-Agent 统一出口
- 重要决策需确认
- 风险及时上报

### 7.3 文档输出

- 需求文档：PM-Agent 负责
- 技术方案：Dev-Agent 负责
- 测试报告：QA-Agent 负责
- 交接文档：任务执行者负责

---

## 八、违规处理

| 违规项 | 处理 |
|--------|------|
| 未交接直接转手 | 强制阻塞，要求补交接文档 |
| 未审查直接合并 | 回滚代码，追责 |
| 跳过质量门禁 | 强制阻塞，要求修复 |
| 未经确认擅自决策 | 警告，要求重新确认 |

## 九、配置说明

### team.json 关键配置

```json
{
  "autonomousMode": {
    "enabled": false  // 禁用自主协作模式
  },
  "teamMode": {
    "enabled": true,
    "autoActivateOnSessionStart": true  // 会话启动自动激活
  },
  "collaboration": {
    "discussionFlow": {
      "enabled": true  // 启用需求讨论流程
    },
    "handoff": {
      "strictMode": true,  // 严格交接模式
      "requireHandoffDocument": true
    },
    "codeReview": {
      "strictMode": true,  // 严格审查模式
      "minReviewers": 2
    }
  },
  "qualityGate": {
    "strictMode": true,
    "blockOnFailure": true
  }
}
```
