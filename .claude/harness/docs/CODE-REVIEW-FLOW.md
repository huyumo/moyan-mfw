# 代码审查流程

## 审查状态流转

```
提交审查 → 审查中 → 通过 → 合并
              ↓
        需要修改（打回）
              ↓
         开发者修改
              ↓
         重新审查 → 通过 → 合并
              ↓
          再次打回（循环）
```

---

## 审查结论

审查者可以给出三种结论：

### ✅ 通过
- 代码质量良好，可以直接合并
- 无需修改

### ⚠️ 有条件通过
- 需要小修改（如注释、命名）
- 修改后无需重新审查
- 开发者自行修改即可

### ❌ 不通过（打回）
- 有需要修复的问题
- 开发者必须修改后重新提交审查
- 审查者重新验收

---

## 开发者处理流程

当审查结论为"不通过"时：

1. **查看审查意见**
   - 阅读 `### ❌ 必须修复` 列表
   - 理解每个问题的原因

2. **回复审查意见**
   - 在审查记录中填写"开发者回复"
   - 标记已修复的问题 `[x]`
   - 说明待讨论的问题

3. **修改代码**
   - 修复所有必须修复的问题
   - 运行自测

4. **重新提交审查**
   - 更新 TASK.md 的 `reviewStatus` 字段
   - 通知审查者重新验收

---

## 审查者验收流程

当开发者回复后：

1. **检查修复情况**
   - 确认所有"必须修复"的问题已修复
   - 运行代码质量检查

2. **给出最终结论**
   - 填写"最终确认"部分
   - 标记"所有问题已修复"
   - 签字确认

3. **通过合并**
   - 更新 reviewStatus 为 approved
   - 允许合并代码

---

## TASK.md 字段说明

```yaml
reviewers: 李四，王五      # 审查者列表
reviewStatus: pending     # 审查状态：pending/revision_required/approved
```

## 审查检查规则

| 场景 | Hook 检查 | 结果 |
|------|----------|------|
| 未指定 reviewers | session-end | ⚠️ 警告 |
| 审查未通过就结束 | session-end | ❌ 阻塞 |
| 有必须修复问题但未回复 | session-end | ❌ 阻塞 |
| 已回复但未修复完 | session-end | ❌ 阻塞 |
| 已修复所有问题 | session-end | ⚠️ 等待审查者确认 |
| 审查者已批准 | session-end | ✅ 通过 |

---

## 示例：打回修改流程

### 第 1 步：审查者提交意见

```markdown
## 审查结论

- [ ] 通过
- [ ] 有条件通过
- [x] 不通过 - 需要修改后重新审查

## 审查意见

### ❌ 必须修复

1. 函数 `calculateTotal` 缺少类型定义
2. 未处理空数组的边界情况
```

### 第 2 步：开发者回复

```markdown
### 开发者回复

**回复人**：李四  
**日期**：2026-04-08 15:30

**已修复的问题**：
- [x] 函数 `calculateTotal` 缺少类型定义 - 已添加返回类型 `number`
- [x] 未处理空数组的边界情况 - 已添加 `if (items.length === 0)` 检查

**待讨论**：
- 无
```

### 第 3 步：审查者确认

```markdown
## 最终确认

**审查人**：王五  
**日期**：2026-04-08 16:00

- [x] 所有问题已修复
- [x] 确认通过
```

---

**审查原则**：
- 对事不对人
- 问题要具体、可执行
- 及时回复，避免阻塞
- 重大问题必须修复，小问题可以有条件通过
