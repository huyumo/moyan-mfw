# 测试策略流程说明

> 目的：防止"边写边改"的低效循环，确保测试编写前完成技术对齐

---

## 问题背景

之前遇到的问题：

| 问题 | 表现 | 根本原因 |
|------|------|----------|
| 双击不触发 | `dblclick()` 无效 | 前端用 `touchstart`，测试用鼠标事件 |
| 页面跳转超时 | `waitForURL('/pages/...')` 失败 | uni-app Hash 路由是 `#/pages/...` |
| 输入框 fill 失败 | `uni-input` 不支持 fill | 自定义组件需要用 `type()` |
| 端口访问空白 | 配置 5173，实际 5174 | 测试人员不知道 uni-app 端口会变化 |
| 选择器不稳定 | 依赖 CSS 类名 | 开发未提供 `data-testid` |

**这些都是可以避免的问题** —— 如果在测试编写前完成技术对齐。

---

## 新流程

```
┌─────────────────────────────────────────────────────────────────┐
│                    测试开发流程（改进后）                        │
└─────────────────────────────────────────────────────────────────┘

1. PM 组织技术对齐会议
   ├─ 开发讲解技术栈（框架、路由、事件处理）
   ├─ 测试提出问题和关注点
   └─ 共同填写《技术对齐会议记录》

2. 开发添加测试友好的选择器
   ├─ 为关键元素添加 data-testid
   └─ 测试验证选择器可用

3. 测试编写测试策略文档
   ├─ 记录技术栈理解
   ├─ 记录测试方法
   └─ 记录配置信息

4. 运行 hook:test-strategy 验证
   ├─ 检查测试策略文档是否存在
   └─ 检查基本结构是否完整

5. 编写第一个验证测试
   ├─ 验证事件模拟方法正确
   ├─ 验证选择器可用
   └─ 验证路由等待有效

6. 开始编写完整测试


┌─────────────────────────────────────────────────────────────────┐
│  Hook 集成点：hook:test-strategy                                 │
│  - 在 pre-code 阶段运行                                          │
│  - 检测任务是否涉及测试                                          │
│  - 如果有测试，强制要求测试策略文档                              │
│  - 没有策略文档 → 阻塞编码                                       │
└─────────────────────────────────────────────────────────────────┘
```

---

## 角色职责

### PM（项目经理）
- [ ] 组织技术对齐会议
- [ ] 确保开发 + 测试都参会
- [ ] 跟踪《技术对齐会议记录》填写
- [ ] 确认第一个验证测试通过

### 开发
- [ ] 讲解技术栈（框架、路由、事件处理）
- [ ] 为关键元素添加 `data-testid`
- [ ] 回答测试关于组件特性的问题
- [ ] 确认配置信息（端口、路由模式等）

### 测试
- [ ] 提出问题和关注点
- [ ] 编写《测试策略文档》
- [ ] 编写第一个验证测试
- [ ] 验证方法论正确后再编写完整测试

## Harness 命令

```bash
# 运行测试策略检查
pnpm run hook:test-strategy

# 生成测试模板（支持多种框架）
pnpm run generate:test-template --framework=uni-app --output=tests/e2e/sample.spec.ts
pnpm run generate:test-template --framework=vue --output=tests/e2e/sample.spec.ts
pnpm run generate:test-template --framework=react --output=tests/e2e/sample.spec.ts
pnpm run generate:test-template --framework=list  # 查看支持的框架

# 运行 pre-code 检查（包含 test-strategy 检查）
pnpm run hook:pre-code
```

## 检查清单

### 技术对齐会议检查清单

- [ ] 开发讲解了前端框架
- [ ] 确认了路由模式（Hash/History）
- [ ] 确认了开发服务器端口
- [ ] 确认了事件处理方式（click/touchstart）
- [ ] 确认了表单组件的测试方法
- [ ] 开发承诺添加 data-testid 的元素清单
- [ ] 测试提出了所有疑问并得到解答

### 测试策略文档检查清单

- [ ] 技术栈说明填写完整
- [ ] 测试方法填写完整
- [ ] 配置信息填写完整
- [ ] 已知问题和解决方案记录
- [ ] 测试覆盖范围明确

### 第一个验证测试检查清单

- [ ] 元素选择器可用
- [ ] 事件模拟方法正确
- [ ] 异步等待有效
- [ ] 路由跳转正确
- [ ] PM/开发确认通过

## 常见问题解答

### Q1: 小项目也需要这个流程吗？

**答**：取决于项目复杂度。
- 如果测试人员已经非常熟悉项目技术栈，可以简化流程
- 但《测试策略文档》是必须的（可以简短，但不能没有）

### Q2: 已经写了一半的测试怎么办？

**答**：
1. 立即停止编写新测试
2. 补充技术对齐会议
3. 补充测试策略文档
4. 验证现有测试方法论是否正确
5. 如不正确，修正后继续

### Q3: 技术对齐会议记录是必须的吗？

**答**：《技术对齐会议记录》是**建议**，但《测试策略文档》是**强制**。
- Hook 会检查测试策略文档是否存在（强制）
- 技术对齐记录是预警（有最好，没有会警告但不会阻塞）

### Q4: 如何判断任务是否涉及测试？

**答**：Hook 会自动检测 TASK.md 内容，匹配以下关键词：
- 测试、e2e、playwright、unit test、vitest
- spec.ts、test 文件、编写测试、修复测试
