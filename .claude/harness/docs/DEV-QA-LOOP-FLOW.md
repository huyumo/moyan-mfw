# 单任务流工作模式 - 开发测试循环 (Dev-QA Loop)

> **状态**: 强制执行 | **优先级**: 最高

---

## 核心原则

### 1. 任务细化原则

**每个任务必须是独立的、可交付的小单元**，具体标准：

| ✅ 正确示例 | ❌ 错误示例 |
|------------|------------|
| "实现用户列表页的搜索功能" | "开发用户管理模块" |
| "添加角色权限分配按钮" | "完成 RBAC 权限系统" |
| "修复搜索框清除按钮不响应问题" | "修复所有已知 bug" |
| "编写用户列表页 E2E 测试" | "完成所有测试" |

**细化标准**：
- 一个任务对应**一个具体的功能点**或**一个按钮交互**
- 开发任务：完成时间 ≤ 2 小时
- 测试任务：覆盖单一场景/页面
- 任务描述必须明确**验收标准**

## 工作流程（开发 - 测试循环）

### 步骤 1: 任务拆解 (PM/主 Agent)

```
需求 → 功能清单 → 任务列表 → TASK.md
```

**功能清单模板**：

```markdown
## 功能清单

### 页面：用户管理

| 序号 | 功能点 | 类型 | 复杂度 | 验收标准 |
|------|--------|------|--------|----------|
| 1 | 用户列表展示 | 开发 | P2 | 显示用户表格，包含分页 |
| 2 | 搜索功能 | 开发 | P2 | 支持用户名模糊搜索 |
| 3 | 搜索清除按钮 | 开发 | P3 | 点击清除搜索条件并刷新 |
| 4 | 列表页 E2E 测试 | 测试 | P2 | 覆盖列表加载、搜索场景 |
```

---

### 步骤 2: 任务领取 (子 Agent)

**单任务锁机制**：

```
[pending] → 领取 → [in_progress] → 开发完成 → [awaiting_qa] → QA 测试
                                                           ↓
                                                        通过？
                                                           ↓ 是              ↓ 否
[提交代码] ←─ [completed] ←─ 更新状态 ←─ [qa_approved] ←─┘         [rejected]
   ↑                                                                     ↓
   └─────────────────────────────────────────────────────────────→ 打回修复
```

**任务状态流转**：

| 状态 | 说明 | 下一步 | 责任人 |
|------|------|--------|--------|
| `pending` | 待处理 | → in_progress | 任意 Agent |
| `in_progress` | 开发中 | → awaiting_qa | Developer |
| `awaiting_qa` | 待测试 | → qa_approved/rejected | QA-Agent |
| `qa_approved` | 测试通过 | → completed | QA-Agent |
| `rejected` | 打回修复 | → in_progress | Developer |
| `completed` | 已完成 | → 提交归档 | QA-Agent |

**规则**：
- 子 Agent **每次只能领取 1 个任务**
- 任务状态必须是 `pending` 才能领取
- **开发完成后不能直接提交** → 必须进入 `awaiting_qa` 等待测试
- **QA 测试通过后** → 由 QA-Agent 提交代码
- **测试不通过** → 打回给开发修复，状态改回 `in_progress`

---

### 步骤 3: 任务执行

#### 3.1 开发任务流程

```
读取 TASK.md
  ↓
需求分析 (hooks/task-analysis)
  ↓
技术方案 (hooks/pre-code)
  ↓
编码实现
  ↓
自测验证
  ↓
【关键】提交 QA 审查 → 状态改为 awaiting_qa
  ↓
等待 QA 测试结果...
  ↓
  ├─ 测试通过 → 更新状态为 qa_approved → QA 提交代码 → completed
  └─ 测试失败 → 接收缺陷报告 → 修复问题 → 重新提交 QA
```

#### 3.2 QA 测试流程

```
读取 awaiting_qa 任务
  ↓
审查代码变更
  ↓
执行测试（单元/E2E/手动）
  ↓
记录测试结果
  ↓
  ├─ 测试通过 → 更新状态为 qa_approved → 提交代码 → completed
  └─ 测试失败 → 更新状态为 rejected → 编写缺陷报告 → 打回开发
```

---

### 步骤 4: 提交检查 (hooks/code-quality-gate)

**强制检查项**：

| 检查项 | 说明 | 失败处理 |
|--------|------|----------|
| 类型检查 | `pnpm typecheck` 通过 | 阻塞 |
| 自测报告 | `self-test-report.md` 存在 | 阻塞 |
| QA 审批 | 状态为 `qa_approved` | **阻塞 - 关键** |
| 功能标记 | 无"开发中"、"TODO"等标记 | 阻塞 |

**新增：QA 审批检查**
- 只有 QA-Agent 审批通过的任务才能提交
- 未通过 QA 的提交会被 hooks 拒绝

### 缺陷处理流程

当 QA 发现缺陷时：

1. **QA 操作**：
   - 更新任务状态：`awaiting_qa` → `rejected`
   - 编写缺陷报告（缺陷描述、复现步骤、期望行为）
   - @开发人员

2. **开发操作**：
   - 接收缺陷通知
   - 阅读缺陷报告
   - 修复问题
   - 重新自测
   - 提交 QA 审查（状态改回 `awaiting_qa`）

3. **循环继续**：
   - QA 重新测试
   - 通过 → 提交
   - 失败 → 再次打回

## 角色职责

### 开发人员 (Backend-Dev, Frontend-Dev)

**职责**：
- ✅ 领取一个 `pending` 状态的任务
- ✅ 开发完成后进行自测
- ✅ 提交 QA 审查（状态改为 `awaiting_qa`）
- ✅ 根据缺陷报告修复问题
- ❌ **禁止直接提交代码**（必须由 QA 提交）

### 测试人员 (QA-Agent)

**职责**：
- ✅ 审查 `awaiting_qa` 状态的任务
- ✅ 执行代码审查和测试
- ✅ 批准通过的任务（→ `qa_approved`）
- ✅ 打回失败的任务（→ `rejected`）并编写缺陷报告
- ✅ **负责提交通过的代码**（关键职责）

### 项目经理 (PM-Agent)

**职责**：
- ✅ 任务拆解和分配
- ✅ 跟踪任务进度
- ✅ 协调开发 - 测试循环
- ✅ 任务归档

## 任务状态定义

| 状态 | 说明 | 可领取 | 可提交 |
|------|------|--------|--------|
| `pending` | 待处理 | ✅ 是 | ❌ 否 |
| `in_progress` | 开发中 | ❌ 否 | ❌ 否 |
| `awaiting_qa` | 待测试 | ❌ 否 | ❌ 否 |
| `qa_approved` | 测试通过 | ❌ 否 | ✅ 是 (QA) |
| `rejected` | 打回修复 | ❌ 否 | ❌ 否 |
| `completed` | 已完成 | ❌ 否 | ✅ 已提交 |

---

## 钩子检查点

| 钩子名称 | 触发时机 | 检查内容 |
|----------|----------|----------|
| `session-start` | 会话开始 | 任务状态、会话锁 |
| `task-analysis` | 任务开始前 | 需求分析完整性 |
| `meeting-required` | 复杂任务 | 是否需要多 Agent 协作 |
| `pre-code` | 编码前 | 技术方案、边界条件 |
| `code-quality-gate` | 提交前 | 类型检查、自测报告、**QA 审批状态** |
| `qa-review` | 开发完成后 | 代码审查、测试执行 |
| `session-end` | 会话结束 | 任务状态更新、归档 |

---

## 违规处理

| 违规行为 | 处理措施 |
|----------|----------|
| 开发跳过 QA 直接提交 | hooks 拒绝，强制退回 |
| 开发自测未通过 | 拒绝提交 QA，退回修复 |
| QA 未测试直接批准 | 记录违规，重新审查 |
| QA 测试通过不提交 | 提醒提交，记录产出 |
| 虚假测试报告 | 记录违规，强制重新执行 |
