🔍 混乱根因 vs 新方案 — 为什么不会重蹈覆辙

同样是"有多个文件"，但原因和结果完全不同

🔴 当前混乱的根因

根因 1：依赖全混在一起

一个 package.json 同时声明 NestJS + Vue + Vite + ElementPlus
→ pnpm install 时全部装到一个 node_modules
→ 编译时所有三方库都可见

根因 2：tsconfig 交叉继承

tsconfig.shared.json 继承 tsconfig.backend.json
→ shared 被迫用 CJS / node 解析
→ frontend 消费 shared 时被迫绕道 dist/shared

根因 3：模块体系互撞

backend 要 CJS / node 解析
frontend 要 ESNext / bundler 解析
→ 两个世界观硬塞进同一个 node_modules

🟢 新方案：隔离但统一

解决 1：依赖彻底隔离

每个子层 package.json 只声明自己的依赖
backend: 只有 NestJS / TypeORM
frontend: 只有 Vue / Vite / ElementPlus
shared: 零运行时依赖
→ 编译时各自只看到自己的依赖

解决 2：tsconfig 互不继承

每个子层 tsconfig.json 独立配置，只继承 workspace 根
→ backend: module=CJS, moduleResolution=node
→ frontend: module=ESNext, moduleResolution=bundler
→ shared: 独立编译，CJS+ESM 双输出
→ 互不干扰

解决 3：模块体系互不碰撞

每个子层在自己的 moduleResolution 下工作
→ CJS 和 ESNext 永远不在同一个编译上下文中

📦 npm 发布视角：消费者看到什么？

内部子层的 package.json 全部标记 "private": true，永远不发布

消费者侧：
$ npm i moyan-mfw-base ← 只装一个包

导入方式：
import { createBaseBackendApp } from 'moyan-mfw-base/backend'
import { createBaseAdminApp } from 'moyan-mfw-base/frontend'
import { AppDto } from 'moyan-mfw-base/shared'

npm 包内文件结构：
moyan-mfw-base/
├── package.json ← exports: {"./backend":..., "./frontend":..., "./shared":...}
├── dist/backend/index.js ← 编译后的 CJS 产物
├── dist/frontend/index.mjs ← 编译后的 ESM 产物
├── dist/shared/index.js ← 编译后的共享层
└── migrations/ ← 数据库迁移

❌ 子层 package.json 不发布 → 消费者完全感知不到
❌ tsconfig.json 不发布
❌ src/ 源码不发布（只有 dist/）

✅ 结论：当前混乱 = 架构缺陷，新方案 = 架构修复

当前是"一个 package.json 管三层"的架构缺陷，导致依赖和编译配置互相污染。
新方案是"三层各自自治 + 根层统一导出"的架构修复，子层互不干扰，对外仍是一个包。

这不是"多几个文件"的问题，而是模块边界是否清晰的问题。

🎯 下一个问题：子层是否要注册到 pnpm workspace？

当前 pnpm-workspace.yaml 是显式路径，不是通配 glob

A

不注册子层，保持最简

pnpm-workspace.yaml 不变，子层 package.json 仅供 IDE 和构建脚本内部使用。所有依赖仍由根 package.json 统一声明（通过 pnpm catalog 去重）。构建脚本串联执行 shared → backend → frontend

优点

  • workspace 配置零改动
  • 无 hoist 行为变化
  • 改动最小

缺点

  • 子 package.json 的依赖声明只是"文档"，不参与实际解析
  • 根 package.json 仍需声明所有三方依赖
B

注册子层到 workspace，真正依赖隔离

pnpm-workspace.yaml 新增 packages/base/src/backend 等条目。每个子包独立安装自己的依赖，通过 workspace:* 互相引用。根 package.json 仅保留 devDependencies 和构建脚本

优点

  • 真正依赖隔离，backend 看不到 Vue
  • 每个子包可独立 lint/build

缺点

  • workspace 配置膨胀
  • 需解决跨子包引用(src 路径变更)
  • pnpm install 行为变化需验证
