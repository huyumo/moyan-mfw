核心问题：单一 package.json 强迫三层共享依赖，5+ 个 tsconfig 交叉引用冲突不断

🔴 当前结构（packages/base 为例）

packages/base/ — 单一 package.json 管所有
packages/base/
├── package.json ← NestJS + Vue + Vite 全混一起
├── tsconfig.json ← Project References 入口
├── tsconfig.backend.json ← CJS / node 解析
├── tsconfig.frontend.json← ESNext / bundler 解析
├── tsconfig.shared.json ← 继承 backend (被迫 CJS)
├── tsconfig.test.json ← 又一套 paths 映射
├── nest-cli.json
├── vite.config.mts
└── src/
├── backend/ ← NestJS 代码
├── frontend/ ← Vue3 代码
└── shared/ ← 纯类型/常量

packages/base/
├── package.json ← 统一导出：backend/ → frontend/ + shared/

│ ├── backend/

│ │ ├── package.json ← 只依赖 Vue / Vite / ElementPlus
│ │ └── tsconfig.json ← ESNext / bundler，仅管自己

第一个问题：子包的注册方式？

A

pnpm workspace 子包注册

每个子包的 package.json 独立注册到 pnpm-workspace.yaml，通过 moyan-mfw-base-backend 等独立包名引用。根 package.json 用 workspace:* 聚合导出

优点

  • 依赖完全隔离
  • 编译互不干扰
  • IDE 类型解析最清晰

缺点

  • 需要注册到 workspace root
  • 包名变多（base-backend / base-frontend / base-shared）
B

package.json exports 子路径

保持单一包名，通过 exports 字段区分 moyan-mfw-base/backend。子层用独立的 package.json 管理依赖，但不注册到 pnpm workspace

优点

  • 对外包名不变
  • workspace 配置改动最小

缺点

  • TypeScript 路径解析仍需 paths 映射
  • 子层 package.json 的依赖可能被根层 hoist 合并
C

混合模式：exports + npm workspaces

根目录 package.json 用 npm workspaces 声明子层，同时保留 exports 子路径。子层独立 tsconfig + package.json，构建时各自编译到 dist 子目录

优点

  • 兼顾隔离与统一导出
  • 构建脚本简洁

缺点

  • workspace 嵌套层级深
  • pnpm hoist 行为需要精细控制
