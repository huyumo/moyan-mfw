问题 1：本地开发的模块引用路径会变吗？

✅ 包内相对路径：完全不变

相对路径是文件系统层面的距离，和 tsconfig 放在哪里无关。
例如 modules/sys/app/service/ → common/ 永远是 ../../../../common/

✅ 路径别名 @/* ：小幅调整

当前后端 tsconfig: "@/*": ["src/backend/*"]
新 tsconfig 放在 src/backend/ 下: "@/*": ["./*"]（仅改映射值，import 语句不动）

⚠️ 跨包引用：需要区分 dev/prod

内部 dev 开发时：backend → shared 通过 workspace 协议 "@internal/base-shared": "workspace:*"
npm 消费时：通过 exports 子路径 "moyan-mfw-base/shared"
当前代码中的 import ... from 'moyan-mfw-base/shared' 在 dev 时通过 tsconfig paths 映射到源码

🎯 dev 时跨包引用方案：export 条件 + tsconfig paths

最关键的决策：开发时"moyan-mfw-base/shared" 怎么解析？

A

tsconfig paths 指向 src（推荐）

consumer 包的 tsconfig 中用 paths 将 "moyan-mfw-base/shared" 映射到 "../../base/src/shared/index.ts"。npm 消费时走 exports → dist

优点

  • import 语句一行不改
  • dev 即时更新，无构建等待
  • 业界常见做法（Vite/Nx 等）

缺点

  • 新增 consumer 包需手动配置 paths
  • paths 是 tsconfig 特性，非标准 JS 解析
B

统一用 workspace 内部引用（彻底改）

所有 import 改为 '@internal/base-shared'，利用 pnpm workspace 的 node_modules 软链接。npm 发布时不做任何处理（消费者不安装内部包）

优点

  • 纯标准 JS 解析，无 tsconfig hack
  • 编译时能严格校验依赖边界

缺点

  • 大量 import 语句需修改（~200+ 处）
  • 源码和发布产物的 import 不一致
