✅ 方案选定：exports 子路径 + 私有子层 package.json

一个逻辑单元 = 一个 npm 包 | 内部三层私有隔离 | 消费者一个 install 搞定

🎯 第二个问题：子层 package.json 如何影响 pnpm workspace？

当前 pnpm-workspace.yaml 直接声明 packages/* 等全局路径。子层加 package.json 后，需要怎么注册？

A

显式声明每个子包路径

在 pnpm-workspace.yaml 中逐一声明：packages/base/src/backend, packages/base/src/frontend, packages/base/src/shared；扩展包同理。每个子包的名字如 @internal/base-backend 加 @internal scope 防止误发布

优点

  • 依赖完全隔离，hoist 可控
  • 每个子包编译脚本独立
  • @internal scope 防止 npm publish 误操作

缺点

  • workspace 配置膨胀（每个扩展 +3 条）
  • 现有 import 路径需要批量替换
B

全局 glob + .npmignore 排除

pnpm-workspace.yaml 保持现有 packages/** 不变（自动发现所有 package.json）。子层 package.json 标记 private，通过根 package.json 的 files 字段控制发布内容

优点

  • pnpm-workspace.yaml 无需改动
  • import 路径可保持相对路径，改动最小
  • 新增扩展包零配置

缺点

  • hoist 行为由 pnpm 自动决策，依赖隔离不如 A
  • 所有包共享同一个 node_modules 提升
