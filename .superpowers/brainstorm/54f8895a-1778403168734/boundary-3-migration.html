🔴 #3：扩展包数据库迁移文件 — 谁执行？何时执行？

扩展包有自己的 TypeORM entity（如 ext_ad_placements）。表结构的创建和维护谁负责？

方案 A：业务层统一管理迁移

扩展包发布时不带 migration。业务层安装后，框架自动检测新实体并为它们生成 migration 到业务层目录。

类比：Django 的 makemigrations my_app

优点：业务层完全掌控数据库变更；迁移文件在业务层版本控制中

缺点：需要框架层提供 ext:migrate:generate 命令

方案 B：扩展包自带 migration，框架启动时执行

扩展包发布时包含 migration 文件。业务层启动时，框架依次执行所有扩展包的迁移。

优点：扩展包开发者编写迁移，最了解表结构

缺点：多个扩展包迁移顺序和冲突管理复杂；业务层失去控制权

方案 C：开发期 synchronize，生产期手动迁移

扩展包只提供 entity 定义。dev 环境用 synchronize: true 自动建表。

生产部署时，业务层手动写 migration。

缺点：生产运维负担重，容易遗漏

📌 推荐方案 B 改良版：扩展包自带 + 框架编排

扩展包目录结构增加：

extension-ad/
├── migrations/
│   └── 001-create-ad-tables.ts

框架启动时：读所有扩展包的 migration → 按时间戳排序 → 依次执行。

TypeORM 原生支持多来源 migration，框架只需指定路径列表即可。
