问题 1：命名空间由谁决定？

扩展包默认声明 vs 业务层覆盖 vs 映射表

A

方案 A：扩展包声明，不可覆盖

extension.json 写死 routePrefix: "/ext/ad"

简单粗暴。所有扩展包的路径在发布时就固定，业务层只能接受。

风险：两个扩展包买了同一个路径 → 启动报错（只能要求开发者别这么干）

B

方案 B：扩展包声明默认值 + 业务层映射覆盖

扩展包声明 routePrefix: "/ext/ad"（默认）

业务层 main.ts 提供 extensionRouteMap: { "moyan-extension-ad": "/my-ads" }

灵活。业务层有最终决定权。扩展包内部跳转通过 composable 解析实际前缀。

代价：扩展包内部不能用硬编码路径

C

方案 C：路径完全由业务层组装

扩展包只暴露 RouteRecordRaw[]，不带路径假设

业务层在注册路由时指定挂载点

最大控制权。但扩展包内部导航失效（组件不知道自己的路径）

问题 2：扩展包开发如何快速调试？

涉及 API 生成 + 路由构建 + 后端联调三个环节

现状瓶颈

开发一个广告表单：改 DTO → 重启后端 → 刷新 Swagger → 跑 api:build → 重启前端 → 改 Vue

这是"6步循环"，每次改接口都要走一遍

方案：开发期跳过 api:build，用动态代理

开发模式：扩展包写一个 devAdapter.ts，直接用 axios + 类型标注，跳过生成步骤

提交/CI 前：必须跑 api:build 更新生成的代码

效果：DTO 改完 → 重启后端 → 直接改 Vue（3步，少了 Swagger + api:build）
