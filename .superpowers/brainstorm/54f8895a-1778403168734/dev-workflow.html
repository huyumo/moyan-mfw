扩展包开发时如何运行？

方案 B 已确认。核心矛盾：扩展包不是独立应用，必须嵌入业务层启动

方案 A：业务层直接 import，靠 workspace 软链接

开发者在 backend/src/app.modules.ts 临时 import AdModule
在 frontend/src/main.ts 临时 import adRoutes

好处：零额外配置，直接跑

痛点：污染业务层代码，每次换扩展包都要改 main.ts

方案 B：业务层提供 extensions.config.ts 声明式开关

后端：extensions.config.ts 列出启用的扩展包名
前端：同上，createBaseAdminApp 读取并自动加载路由

好处：改一行配置即切换；可提交到 git 作为项目默认

前提：需要框架层支持"按名称动态 import 扩展包"

方案 C：扩展包自带 dev playground

每个扩展包内建最小化的 dev/ 目录
包含独立的 main.ts，仅加载该扩展包

好处：完全隔离，不依赖业务层

代价：维护成本翻倍

推荐：方案 B — 声明式开关

一次配置，随时切换，可提交 git

// backend/src/extensions.config.ts（业务层新增文件）
export const enabledExtensions = ['moyan-extension-ad']

// backend/src/main.ts（框架层需改造以支持自动加载）
import { enabledExtensions } from './extensions.config'
// 框架内部：enabledExtensions 中包含 'moyan-extension-ad'
//   → 自动 import('moyan-extension-ad/backend') → 注入 AdModule + entities + 权限节点

// frontend/src/extensions.config.ts
export const enabledExtensions = ['moyan-extension-ad']
export const extensionRouteMap = {     // ← 方案 B 的命名空间映射
  'moyan-extension-ad': '/ads'        //   业务层可覆盖默认的 /ext/ad
}

关键问题

框架层需要改造吗？ 需要。当前 createBaseBackendApp 不支持"按包名动态 import 扩展包"，但这正是 P0 中 entities/permissionValues 通道的延伸。

这是额外负担还是必要能力？ 必要。没有这个，每个扩展包都要手写 import，扩展包系统无法规模化和自动化。
