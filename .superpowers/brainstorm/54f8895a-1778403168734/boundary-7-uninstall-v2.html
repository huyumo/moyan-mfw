🔴 #7（修正）：安全卸载 + 明确清理

折中方案：框架不自动删，但提供强引导的清理机制，避免残留影响业务系统

A

启动时检测 → 拒绝启动 + 提示清理命令

框架启动时发现：数据库有某扩展包的表，但配置中已移除该扩展包

→ 拒绝启动，终端输出：

⚠️  检测到已卸载扩展包的残留数据
  - moyan-extension-ad (ext_ad_placements, ext_ad_contents, ...)

请运行清理命令后重新启动：
  pnpm ext:cleanup moyan-extension-ad
或跳过检查（危险）：
  EXT_SKIP_CLEANUP_CHECK=1 pnpm dev
B

启动时警告 + 继续运行

只打印警告，不阻止启动。卸载后数据库表仍然保留。

风险：开发者可能忽略警告，残留数据长期积累

📌 推荐方案 A：拒绝启动 + 明确清理路径

逻辑：

  1. 框架启动时扫描所有扩展包的 entity → 对比数据库中的表
  2. 发现"有表但无配置" → 拒绝启动，输出清理命令
  3. 开发者手动执行 pnpm ext:cleanup → 运行扩展包的 onUninstall 中的 DROP/DELETE 逻辑
  4. 清理后重启正常

扩展包的责任：在 migrations/ 中提供 down 函数，cleanup 时执行。

既不自动删数据（安全），也不让残留数据默默影响系统（可控）。
