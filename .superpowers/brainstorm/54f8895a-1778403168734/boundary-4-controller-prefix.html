🔴 #4：扩展包 Controller 路径与业务层 Controller 碰撞

扩展包的 @Controller('ad-placements') 和业务层的某个 Controller 撞了同名路径怎么办？

方案 A：框架强制前缀隔离（推荐）

框架在注册扩展包 Controller 时，自动在前面加 ext/{extName} 全局前缀

效果：ad-placements → 实际路径 /api/ext/ad/ad-placements

优点：零碰撞风险；框架层强制执行，开发者无感知

缺点：需要框架在 NestJS 路由注册时做前缀注入

方案 B：扩展包开发者手动写前缀

约定：Controller 路径统一写 'ext/ad/ad-placements'

优点：零框架改造

缺点：依赖开发者遵守约定；npm 发布后路径固定，业务层无法覆盖

📌 推荐方案 A：框架强制前缀

在 createExtensionBackendApp 和 createBaseBackendApp 中：

// 框架内部逻辑
const extPrefix = extensionName.replace('moyan-extension-', '') // 'ad'
app.setGlobalPrefix(`api/ext/${extPrefix}`)

扩展包开发者只需写 @Controller('ad-placements')，框架自动补全前缀。

与前端路由的命名空间前缀对称设计。
