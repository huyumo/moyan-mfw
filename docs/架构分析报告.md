## 二、后端架构详图

### 2.1 请求处理流水线

```mermaid
sequenceDiagram
    participant Client as 前端应用
    participant CORS as CORS 中间件
    participant AG as AuthGuard
    participant PG as PermissionGuard
    participant VP as ValidationPipe
    participant LI as LoggingInterceptor
    participant CTL as Controller
    participant SVC as Service
    participant AI as AuditInterceptor
    participant TI as TransformInterceptor
    participant EF as AllExceptionsFilter
    participant DB as MySQL

    Client->>CORS: HTTP Request
    CORS->>AG: 验证通过
    AG->>AG: JWT 解析 → 查询用户信息 → 注入 req.user
    AG->>PG: 用户已验证
    PG->>PG: 读取 @RequirePermission / @Permission 元数据
    PG->>PG: 位运算权限检查 (BigInt)
    PG->>VP: 权限通过
    VP->>VP: class-validator 校验 (whitelist + transform)
    VP->>LI: 参数合法
    LI->>LI: 记录请求方法/路径/IP
    LI->>CTL: 进入业务层
    CTL->>SVC: 调用 Service
    SVC->>DB: TypeORM Repository / PaginationX
    DB-->>SVC: 返回数据
    SVC-->>CTL: 返回结果
    CTL-->>AI: 响应
    AI->>AI: 读取 @AuditLog 元数据 → 写入 sys_audit_logs
    AI-->>TI: 审计完成
    TI->>TI: 包装为 { code, data, message, timestamp }
    TI-->>Client: 统一响应格式

    Note over EF: 任何步骤抛异常时兜底，统一错误响应
```

```
                            ┌───────────────────┐

                          读取装饰器元数据

                    ┌────────────────┼────────────────┐
                    │                │                │
              isDeveloper?      isOwner?         普通用户
              → 直接放行       → 全部权限       → 位运算检查

                                          ┌──────────▼──────────┐
                                          │  权限值计算流程       │
                                          │                       │
                                          │ 1. 查用户角色列表      │
                                          │ 2. 查角色权限关联      │
```

## 三、前端架构详图

### 3.1 初始化流程

```mermaid
sequenceDiagram
    participant Entry as main.ts
    participant Factory as createBaseAdminApp()
    participant Store as Pinia Stores
    participant Router as Vue Router
    participant Guard as 路由守卫
    participant API as MoAxios
    participant Server as 后端 /api

    Entry->>Factory: 调用工厂函数(配置项)
    Factory->>Factory: createApp(BaseAdminRoot)
    Factory->>Factory: createPinia()
    Factory->>Store: 初始化 authStore / layoutStore
    Factory->>Factory: setupPlugins(ElementPlus + MoAxios)
    Factory->>Router: createBaseAdminRouter()
    Factory->>Guard: setupRouteGuard()
    Factory->>Factory: mount('#app')

    Note over Guard,Server: 用户首次访问
    Guard->>Guard: 检查白名单 (/login, /install)
    Guard->>Guard: 检查 Token 是否存在
    Guard->>Store: initializeAuth()
    Store->>API: GET /api/auth/userinfo
    API->>Server: 携带 Bearer Token
    Server-->>API: 返回用户信息
    Store->>API: GET /api/auth/apps
    Server-->>API: 返回应用列表
    Store->>Store: autoSelectApp(优先默认应用)
    Store->>API: GET /api/auth/permissions
    Server-->>API: 返回权限菜单
    Store->>Store: 权限菜单 → 更新侧边栏
    Guard->>Guard: 匹配路由权限
```

## 七、整改问题清单

通过深度分析，发现以下需要关注的问题：

### 7.1 权限配置不一致（高）

| 问题 | 详情 |
|------|------|
| 前后端权限值不匹配 | 后端定义 `['上架', '发货', '退款']`，前端定义 `['发货', '充值', '接待', '指派']`，仅 `发货` 是交集 |
| 控制器使用了未定义权限值 | `supplier.controller.ts` 的 `listProfiles` 方法使用了 `'添加'`，该值不在 `BUSINESS_PERMISSION_VALUES` 中 |

**建议**：统一前后端权限值定义，将 `BUSINESS_PERMISSION_VALUES` 抽到 `moyan-shared-dict` 包中共享。

### 7.2 实体建模问题（中）

| 问题 | 文件 | 建议 |
|------|------|------|
| `@ManyToOne` 实现一对一语义 | `supplier-member-profile.entity.ts` | 改用 `@OneToOne` + 显式 `@Column('member_id')` |
| `listSupplierProfiles` 无分页 | `supplier.service.ts` | 使用 `PaginationX + WhereBuilder` |
| 更新操作未做字段白名单 | `supplier.service.ts` | 使用 `repository.merge()` 或字段过滤 |

### 7.3 其他注意项（低）

| 问题 | 详情 |
|------|------|
| JWT Secret 硬编码 | `jwt.config.ts` 使用测试值，生产环境必须修改 |
| Redis 配置但未使用 | 已定义 Redis 配置，但无业务代码实际使用 |
| 业务页面均为占位 | `orders`/`reports`/`overview` 等页面等待填充真实内容 |
| 字典与迁移分离 | `business-dict` 定义了字典结构，但表创建依赖独立 migrate 脚本 |
