## 4.1 实体设计规范

### 4.1.1 实体命名规范

| 类型 | 命名规则 | 示例 | 等级 |
|------|----------|------|------|
| 表名 | 小写 + 下划线 + 复数 | `sys_users`, `sys_roles` | 🔴 |
| 主键 | `id` | `id` | 🔴 |
| 外键 | `关联表名单数_id` | `user_id`, `role_id` | 🔴 |
| 字段名 | 小写 + 下划线 | `user_name`, `created_at` | 🔴 |
| 索引 | `idx_表名_字段名` | `idx_users_username` | 🟢 |
| 唯一索引 | `uk_表名_字段名` | `uk_users_username` | 🟢 |
| 外键约束 | `fk_表名_关联表名` | `fk_users_roles` | 🟢 |

### 分表分库命名规范

**水平分表命名（按数据量分表）：**

```
格式：{原表名}_{分表标识}

示例：
- sys_users_0000, sys_users_0001, ... sys_users_9999    # 按 ID 尾号分表（10000 个表）
- sys_orders_2024, sys_orders_2025, ...                 # 按年份分表
- sys_logs_01, sys_logs_02, ... sys_logs_12             # 按月份分表
```

**垂直分表命名（按字段拆分）：**

```
格式：{原表名}_{扩展标识}

示例：
- sys_users_base      # 用户基础信息表（id, username, password 等核心字段）
- sys_users_extend    # 用户扩展信息表（user_id, preferences, settings 等）
- sys_users_profile   # 用户 Profile 表（user_id, avatar, bio, social_links 等）
```

**分库命名（按业务/功能拆分）：**

```
格式：{业务前缀}_db

示例：
- user_db, order_db, product_db      # 按业务分库
- mfw_core_db, mfw_log_db            # 按功能分库
```

| 类型 | 命名规则 | 示例 | 等级 |
|------|----------|------|------|
| 水平分表 | `{原表名}_{序号/日期}` | `sys_users_0000`, `sys_orders_2024` | 🟢 |
| 垂直分表 | `{原表名}_{base/extend/profile}` | `sys_users_base`, `sys_users_extend` | 🟢 |
| 分库 | `{业务前缀}_db` | `user_db`, `order_db` | 🟢 |

**🔴 强制：**
- 分表规则必须在迁移文件中明确定义
- 分表键 (Sharding Key) 必须在 ER 图中标注
- 禁止在代码中硬编码分表表名，必须使用路由中间件（如 TypeORM Sharding）

### 4.1.2 通用字段规范

> **🔴 强制：TypeORM 注解规范详见 [01-编码基础规范.md](./01-编码基础规范.md#113-1-实体类 typeorm-注解规范)**
>
> - `@Column({})` 中必须编写 `comment` 和 `type`
> - `comment` 格式：`{字段名称} - {用途说明}，{业务规则/约束条件}`
> - 如有枚举值，必须列出所有取值及含义

所有业务表必须包含以下审计字段：

```sql
-- 创建时间（自动填充）
created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP

-- 更新时间（自动更新）
updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP

-- 创建人 ID
created_by BIGINT COMMENT '创建人 ID'

-- 更新人 ID
updated_by BIGINT COMMENT '更新人 ID'

-- 逻辑删除标记（可选）
deleted_at TIMESTAMP NULL COMMENT '逻辑删除时间'
```

### 4.1.3 实体类定义

```typescript
// entity/User.entity.ts
import {
  Entity,
  PrimaryGeneratedColumn,
  Column,
  CreateDateColumn,
  UpdateDateColumn,
  DeleteDateColumn,
  ManyToMany,
  JoinTable,
} from 'typeorm';

@Entity('sys_users')
export class User {
  @PrimaryGeneratedColumn('uuid')
  id: string;

  @Column({ name: 'username', length: 50, unique: true })
  username: string;

  @Column({ name: 'password_hash', length: 255 })
  passwordHash: string;

  @Column({ name: 'nickname', length: 50, nullable: true })
  nickname?: string;

  @Column({ name: 'email', length: 100, nullable: true })
  email?: string;

  @Column({ name: 'phone', length: 20, nullable: true })
  phone?: string;

  @Column({ name: 'status', type: 'tinyint', default: 1 })
  status: number; // 1-正常，0-禁用

  @CreateDateColumn({ name: 'created_at' })
  createdAt: Date;

  @UpdateDateColumn({ name: 'updated_at' })
  updatedAt: Date;

  @DeleteDateColumn({ name: 'deleted_at' })
  deletedAt?: Date;

  @Column({ name: 'created_by', nullable: true })
  createdBy?: string;

  @Column({ name: 'updated_by', nullable: true })
  updatedBy?: string;

  @ManyToMany(() => Role)
  @JoinTable({
    name: 'sys_user_roles',
    joinColumn: { name: 'user_id' },
    inverseJoinColumn: { name: 'role_id' },
  })
  roles: Role[];
}
```

### 4.1.4 字段类型选择

| 业务场景 | 推荐类型 | 说明 |
|----------|----------|------|
| 主键 ID | `BIGINT` 或 `VARCHAR(36)` (UUID) | UUID 更安全和分布式友好 |
| 金额 | `DECIMAL(10,2)` | 避免浮点数精度问题 |
| 状态/枚举 | `TINYINT` | 节省空间 |
| 短文本 (<100) | `VARCHAR(100)` | 可变长度 |
| 长文本 | `TEXT` | 文章、描述等 |
| 布尔值 | `TINYINT(1)` | 0/1 |
| 时间戳 | `TIMESTAMP` | 自动时区转换 |
| 日期 | `DATE` | 仅日期 |

### 4.1.5 数据库视图规范

**视图使用场景：**

- 简化复杂查询（多表 JOIN、聚合计算）
- 数据权限隔离（行级/列级权限控制）
- 报表统计查询
- 数据脱敏展示

**🔴 强制：视图命名规范**

| 视图类型 | 命名前缀 | 示例 | 说明 |
|----------|----------|------|------|
| 普通视图 | `v_` | `v_user_profiles` | 简单查询视图 |
| 复杂视图 | `vw_` | `vw_order_statistics` | 多表 JOIN/聚合视图 |
| 权限视图 | `v_auth_` | `v_auth_user_dept` | 带权限控制的视图 |
| 物化视图 | `mv_` | `mv_daily_stats` | 物化视图（缓存结果） |

**视图定义规范：**

```sql
-- 普通视图示例
CREATE OR REPLACE VIEW v_user_profiles AS
SELECT
    u.id,
    u.username,
    u.nickname,
    u.email,
    -- 手机号脱敏展示
    CONCAT(LEFT(u.phone, 3), '****', RIGHT(u.phone, 4)) AS masked_phone,
    -- 状态中文描述
    CASE u.status
        WHEN 1 THEN '正常'
        WHEN 0 THEN '禁用'
        ELSE '未知'
    END AS status_text
FROM sys_users u
WHERE u.deleted_at IS NULL;

-- 复杂视图示例（带权限控制）
CREATE OR REPLACE VIEW v_auth_user_dept AS
SELECT
    u.id AS user_id,
    u.username,
    u.nickname,
    d.id AS dept_id,
    d.dept_name,
    -- 数据权限级别
    CASE
        WHEN ur.role_code = 'ADMIN' THEN 1  -- 管理员：全部数据
        WHEN ur.role_code = 'DEPT_MANAGER' THEN 2  -- 部门经理：本部门 + 下级
        WHEN ur.role_code = 'DEPT_STAFF' THEN 3  -- 部门员工：仅本部门
        ELSE 4  -- 普通用户：仅本人
    END AS permission_level
FROM sys_users u
LEFT JOIN sys_user_roles ur ON u.id = ur.user_id
LEFT JOIN sys_depts d ON u.dept_id = d.id
WHERE u.deleted_at IS NULL;
```

**TypeORM 视图实体定义：**

```typescript
// entities/view/user-profile.view.entity.ts

import { User } from '../user.entity';

@ViewEntity('v_user_profiles', { synchronize: false })
export class UserProfileView {
  @ViewColumn()

  @ViewColumn()

  @ViewColumn()
  nickname: string;

  @ViewColumn()

}

// 使用示例
const profiles = await userRepository
  .createQueryBuilder('user')
  .innerJoin('v_user_profiles', 'profile', 'profile.id = user.id')
  .getMany();
```

**🟢 推荐：视图使用最佳实践**

1. **视图更新限制**
   - 避免在视图上进行 UPDATE/DELETE 操作
   - 需要更新时，直接操作基表

2. **物化视图刷新策略**
   ```sql
   -- 定时刷新（PostgreSQL 示例）
   REFRESH MATERIALIZED VIEW CONCURRENTLY mv_daily_stats;

   -- 或通过应用层定时任务刷新
   ```

3. **视图文档**
   - 在迁移文件中注释说明视图用途
   - 在 ER 图中标注视图依赖关系

**🟢 推荐：视图索引优化**

```sql
-- 为视图创建索引（物化视图）
CREATE UNIQUE INDEX idx_mv_stats_date ON mv_daily_stats (stat_date);
CREATE INDEX idx_mv_stats_dept ON mv_daily_stats (dept_id, stat_date);
```

## 4.3 查询优化

### 4.3.1 索引使用原则

**应该创建索引的场景：**
- 主键（自动创建）
- 外键字段
- WHERE 子句中的字段
- ORDER BY 中的字段
- GROUP BY 中的字段
- 联合查询的连接字段

**不应创建索引的场景：**
- 表记录很少（<1000）
- 频繁更新的字段
- 区分度低的字段（如性别）
- 在函数中使用的字段

### 4.3.2 联合索引最左前缀原则

```typescript
// ✅ 推荐：联合索引 (a, b, c)
CREATE INDEX idx_abc ON table_name(a, b, c);

// 可以使用索引的查询
WHERE a = 1                    // ✅ 使用全部
WHERE a = 1 AND b = 2          // ✅ 使用前两个
WHERE a = 1 AND b = 2 AND c = 3 // ✅ 使用全部

// 不能使用索引的查询
WHERE b = 2                    // ❌ 跳过 a
WHERE c = 3                    // ❌ 跳过 a, b
WHERE b = 2 AND c = 3          // ❌ 跳过 a
```

### 4.3.3 避免 N+1 查询

```typescript
// ❌ 避免：N+1 查询问题
const users = await userRepository.find();
for (const user of users) {
  user.roles = await roleRepository.findByUserId(user.id); // N 次查询
}

// ✅ 推荐：使用 JOIN 一次性查询
const users = await userRepository
  .createQueryBuilder('user')
  .leftJoinAndSelect('user.roles', 'role')
  .getMany();
```

### 4.3.4 查询性能分析

```typescript
// 使用 EXPLAIN 分析查询
EXPLAIN SELECT * FROM sys_users WHERE username = 'test';

// 分析结果关注点：
// - type: 访问类型（system > const > eq_ref > ref > range > index > ALL）
// - key: 实际使用的索引
// - rows: 扫描行数
// - Extra: 额外信息（Using index=好，Using filesort=需优化）
```

### 4.3.5 分页查询优化

```typescript
// ❌ 避免：大数据量时使用 OFFSET
const [list, total] = await userRepository.findAndCount({
  skip: 100000,  // 大 OFFSET 性能差
  take: 20,
});

// ✅ 推荐：使用游标分页（基于 ID）
const list = await userRepository
  .createQueryBuilder('user')
  .where('user.id > :lastId', { lastId: 'last-seen-id' })
  .orderBy('user.id', 'ASC')
  .limit(21)  // 多取 1 条判断是否有更多
  .getMany();

const hasNext = list.length > 20;
if (hasNext) list.pop();
```

## 4.4 事务处理

### 4.4.1 事务使用场景

- 多个表的写入操作
- 需要保证数据一致性的操作
- 涉及金额的操作

### 4.4.2 事务实现

```typescript
// ✅ 推荐：使用装饰器事务
@Injectable()
export class UserService {
  @Transactional()
  async register(dto: RegisterDTO): Promise<User> {
    // 1. 创建用户
    const user = await this.userDAO.create({
      username: dto.username,
      password: await bcrypt.hash(dto.password, 10),
    });

    // 2. 绑定默认角色
    const defaultRole = await this.roleDAO.getDefault();
    await this.userDAO.bindRole(user.id, defaultRole.id);

    // 3. 记录审计日志
    await this.auditLogDAO.create({
      action: 'USER_REGISTER',
      userId: user.id,
    });

    return user;
  }
}

// ✅ 推荐：手动事务
async transfer(fromId: string, toId: string, amount: number) {
  const queryRunner = this.dataSource.createQueryRunner();
  await queryRunner.connect();
  await queryRunner.startTransaction();

  try {
    await queryRunner.manager.decrement(
      'accounts',
      { id: fromId },
      'balance',
      amount,
    );
    await queryRunner.manager.increment(
      'accounts',
      { id: toId },
      'balance',
      amount,
    );

    await queryRunner.commitTransaction();
  } catch (error) {
    await queryRunner.rollbackTransaction();
    throw error;
  } finally {
    await queryRunner.release();
  }
}
```

### 4.4.3 事务隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|----------|------|------------|------|
| READ UNCOMMITTED | ✅ | ✅ | ✅ |
| READ COMMITTED (默认) | ❌ | ✅ | ✅ |
| REPEATABLE READ | ❌ | ❌ | ✅ |
| SERIALIZABLE | ❌ | ❌ | ❌ |

## 4.5 数据脱敏

### 4.5.1 脱敏规则

```typescript
// 手机号脱敏
function maskPhone(phone: string): string {
  return phone.replace(/(\d{3})\d{4}(\d{4})/, '$1****$2');
}

// 邮箱脱敏
function maskEmail(email: string): string {
  const [name, domain] = email.split('@');
  return `${name.charAt(0)}****@${domain}`;
}

// 身份证号脱敏
function maskIdCard(idCard: string): string {
  return idCard.replace(/(\d{6})\d{8}(\d{4})/, '$1********$2');
}
```

### 4.5.2 序列化时脱敏

```typescript
// 使用 TypeORM 的 @Transform 装饰器
import { Expose, Transform } from 'class-transformer';

export class UserResponseDTO {
  @Expose()
  id: string;

  @Expose()
  username: string;

  @Expose()
  @Transform(({ value }) => value ? value.replace(/(\d{3})\d{4}(\d{4})/, '$1****$2') : null)
  phone?: string;

  // 不暴露密码
  // passwordHash 不在 DTO 中定义
}
```
