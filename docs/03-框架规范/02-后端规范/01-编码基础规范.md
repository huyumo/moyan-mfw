## 1.1 命名规范

### 1.1.1 文件命名

**🔴 强制：所有 .ts 文件使用 kebab-case（短横线分隔）+ 功能后缀**

| 类型 | 规则 | 示例 | 等级 |
|------|------|------|------|
| 服务类 | `[name].service.ts` | `user.service.ts`, `user-message.service.ts` | 🔴 |
| 控制器 | `[name].controller.ts` | `user.controller.ts`, `auth.controller.ts` | 🔴 |
| 实体 | `[name].entity.ts` | `user.entity.ts`, `role.entity.ts` | 🔴 |
| DTO/VO | `[name].dto.vo.ts` | `user.dto.vo.ts`, `create-user.dto.vo.ts` | 🔴 |
| 类型定义 | `[name]-types.ts` | `user-types.ts`, `common-types.ts` | 🔴 |
| 工具函数 | `[name].ts` 或 `[name]-utils.ts` | `utils.ts`, `string-utils.ts` | 🔴 |
| 配置文件 | `[name].config.ts` | `database.config.ts`, `app.config.ts` | 🔴 |
| 中间件 | `[name].middleware.ts` | `auth.middleware.ts`, `logger.middleware.ts` | 🔴 |
| 常量文件 | `[name].ts` | `error-codes.ts`, `api-constants.ts` | 🔴 |

**命名转换示例：**

| 旧命名（PascalCase） | 新命名（kebab-case） |
|---------------------|---------------------|
| `UserService.ts` | `user.service.ts` |
| `UserMessageService.ts` | `user-message.service.ts` |
| `UserController.ts` | `user.controller.ts` |
| `UserEntity.ts` | `user.entity.ts` |
| `CreateUserDTO.ts` | `create-user.dto.vo.ts` |
| `UserTypes.ts` | `user-types.ts` |
| `StringUtils.ts` | `string-utils.ts` |
| `DatabaseConfig.ts` | `database.config.ts` |
| `AuthMiddleware.ts` | `auth.middleware.ts` |
| `ERROR_CODES.ts` | `error-codes.ts` |

### 1.1.2 目录命名

```
✅ 推荐：
src/controllers/          # 控制器层
src/services/             # 服务层
src/entities/             # 实体层
src/dto/                  # 数据传输对象
src/middleware/           # 中间件
src/utils/                # 工具函数
src/config/               # 配置文件
src/types/                # 类型定义

❌ 避免：
src/Controller/           # 不要大写
src/controller/           # 不要单数（除非语义需要）
src/myUtils/              # 避免个人化命名
```

**🔴 强制：目录名使用小写 + 复数形式**

| 目录 | 正确 | 错误 |
|------|------|------|
| 控制器层 | `controllers/` | `Controller/`, `controller/` |
| 服务层 | `services/` | `Service/`, `service/` |
| 实体层 | `entities/` | `Entity/`, `entity/` |
| 工具函数 | `utils/` | `Utils/`, `util/` |

### 1.1.3 代码命名

**规则：类/接口使用 PascalCase，变量/函数使用 camelCase，常量使用 UPPER_SNAKE_CASE**

```typescript
// 变量 - camelCase
const userName = 'test';
const isLoading = false;
const userList: User[] = [];

// 常量 - UPPER_SNAKE_CASE
const MAX_RETRY_COUNT = 3;
const API_TIMEOUT = 10000;
const DEFAULT_PAGE_SIZE = 10;

// 类/接口 - PascalCase（文件名：kebab-case）
// 文件：user.service.ts
class UserService {
  async getUserById(id: string): Promise<User> {
    // ...
  }
}

// 文件：user.entity.ts
interface UserInfo {
  id: string;
  name: string;
  email: string;
}

// 枚举 - PascalCase（文件名：kebab-case）
// 文件：user-status.enum.ts
enum UserStatus {
  ACTIVE = 'active',
  INACTIVE = 'inactive',
  DISABLED = 'disabled',
}

// 泛型类型 - PascalCase
interface Response<T = any> {
  code: number;
  data: T;
  message?: string;
}
```

**类名与文件名对应关系：**

| 文件名 | 类名 | 说明 |
|--------|------|------|
| `user.service.ts` | `class UserService` | 类名 = 模块名 PascalCase + 功能后缀 |
| `user-message.service.ts` | `class UserMessageService` | 子模块名保持驼峰 |
| `user.controller.ts` | `class UserController` | 控制器类 |
| `user.entity.ts` | `class User` 或 `class UserEntity` | 实体类 |
| `create-user.dto.vo.ts` | `class CreateUserDto` | DTO 类 |

> **🔴 强制：实体类 TypeORM 注解规范详见 [04-数据库规范.md](./04-数据库规范.md#413-实体类-typeorm-注解规范)**
>
> - `@Column({})` 中必须编写 `comment` 和 `type`
> - `comment` 格式：`字段名称 - 用途说明 - 约束/规则`
> - 有枚举值时必须列出所有取值及含义

### 1.1.4 数据库相关命名

```typescript
// 表名 - 小写 + 下划线 + 复数
// ✅ 推荐：

// ❌ 避免：

// 外键约束 - fk_表名_关联表

```

**模块划分原则：**

| 原则 | 说明 | 示例 |
|------|------|------|
| 按业务领域 | 每个业务模块独立成目录 | `sys/`, `business/`, `crm/` |

| 边界清晰 | 模块间通过接口通信，禁止跨子模块直接依赖 | `user.service.ts` 不直接导入 `permission.service.ts` |

### 1.2.3 模块依赖方向规则

**🔴 强制：禁止循环依赖和逆向依赖**

```
模块内依赖方向：
controllers/ (控制器层)
  ↓
services/ (服务层)
  ↓
entities/ (实体层)
  ↓
utils/ (工具层)
```

**模块间依赖规则：**

```
✅ 推荐：
- 同一模块内的子模块可直接依赖（如 sys/app → sys/user）
- 跨模块依赖通过 common/ 接口解耦
- 公共模块可被所有业务模块依赖

❌ 避免：
- 业务模块 → common/内部子模块（应通过 index.ts 统一导出）
- 子模块间循环依赖（如 sys/app → sys/permission → sys/app）
- entities/ → services/（逆向依赖）
```

**依赖规则说明：**

| 规则 | 说明 | 示例 |
|------|------|------|
| 单向依赖 | 上层可依赖下层，下层不可依赖上层 | `sys/user/services` → `sys/user/entities` ✅ |
| 禁止循环 | 模块间禁止循环依赖 | `sys/app` → `sys/user` → `sys/app` ❌ |
| 公共优先 | 公共工具/类型可被所有层依赖 | `common/utils` → 所有层 ✅ |
| 接口解耦 | 跨模块依赖通过接口解耦 | 使用 `IUserService` 而非直接依赖 `UserService` |
| 模块内自由 | 同一子模块内可直接依赖 | `sys/app/controllers` → `sys/app/services` ✅ |

**循环依赖解决方案：**

```typescript
// ❌ 错误：循环依赖
// modules/sys/user/services/user.service.ts
import { AppService } from '../../app/services/app.service';

// modules/sys/app/services/app.service.ts
import { UserService } from '../../user/services/user.service';

// ✅ 正确：通过接口解耦
// modules/sys/user/services/user.service.ts
import { IAppService } from '../../app/services/app.service.interface';

export class UserService {
  constructor(
    @Inject('IAppService')
    private appService: IAppService
  ) {}
}

// modules/sys/app/services/app.service.interface.ts
export interface IAppService {
  getAppById(id: string): Promise<App>;
}

// modules/sys/app/services/app.service.ts
import { IAppService } from './app.service.interface';

export class AppService implements IAppService {
  async getAppById(id: string): Promise<App> {
    // ...
  }
}
```

### 1.6.1 自定义错误类

### 1.6.2 Try-Catch 规范

**🔴 强制：业务代码中尽量避免使用 try-catch，使用全局异常过滤器统一管理**

**原因：**
1. **错误日志统一**：所有异常集中记录，格式统一，便于排查
2. **代码简洁**：业务代码不需要到处写 try-catch，专注于业务逻辑
3. **错误分类处理**：根据异常类型自动返回不同的 HTTP 状态码
4. **避免遗漏**：不会因为忘记写 try-catch 导致错误被吞掉

---

#### 全局异常过滤器示例

```typescript
// common/filters/app.filter.ts
import {
  ExceptionFilter,
  Catch,
  ArgumentsHost,
  HttpException,
  HttpStatus,
  Logger,
} from '@nestjs/common';
import { Request, Response } from 'express';

/**
 * 全局异常过滤器
 * 统一处理所有未被捕获的异常，返回标准错误响应格式
 */
@Catch()
export class AppFilter implements ExceptionFilter {
  private readonly logger = new Logger(AppFilter.name);

  catch(exception: unknown, host: ArgumentsHost) {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse<Response>();
    const request = ctx.getRequest<Request>();

    // 判断异常类型
    let status: number;
    let message: string;
    let error: string;

    if (exception instanceof HttpException) {
      // HTTP 异常
      status = exception.getStatus();
      const exceptionResponse = exception.getResponse();
      message =
        typeof exceptionResponse === 'string'
          ? exceptionResponse
          : (exceptionResponse as any).message;
      error = exception.name;
    } else if (exception instanceof Error) {
      // 普通错误
      status = HttpStatus.INTERNAL_SERVER_ERROR;
      message = exception.message;
      error = exception.name;
    } else {
      // 未知错误
      status = HttpStatus.INTERNAL_SERVER_ERROR;
      message = 'Internal server error';
      error = 'UnknownError';
    }

    // 记录错误日志
    if (status >= 500) {
      // 服务器错误，记录完整堆栈
      this.logger.error(
        `[${request.method}] ${request.url}`,
        exception instanceof Error ? exception.stack : '',
      );
    } else {
      // 客户端错误，记录简要信息
      this.logger.warn(
        `[${request.method}] ${request.url} - ${status}: ${message}`,
      );
    }

    // 返回统一错误响应格式
    response.status(status).json({
      code: status,
      message,
      error,
      timestamp: new Date().toISOString(),
      path: request.url,
    });
  }
}
```
