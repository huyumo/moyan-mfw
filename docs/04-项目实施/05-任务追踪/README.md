## 双层任务记录体系

### 轻量级：`.claude/TASK.md`

**用途**：记录当前活跃任务状态，供会话开始快速读取

**特点**：
- 单一文件，快速更新
- 记录当前任务、状态、下一步行动
- 包含会话锁机制，防止多会话冲突

**文件头格式**：
```yaml
---
task: 任务名称
status: pending | in_progress | blocked | completed
priority: P0 | P1 | P2 | P3
started: YYYY-MM-DD
updated: YYYY-MM-DD HH:mm
session: 会话 ID
lock: 时间戳（可选）
assignee: @角色
---
```

**必选章节**：
- 当前目标 - 一句话描述
- 已完成 - 完成项目列表
- 进行中 - 进行中的任务（含状态、阻塞点）
- 待开始 - 待开始任务（标注优先级）
- 相关文件 - 文件路径列表

---

### 重量级：`docs/04-项目实施/05-任务追踪/[编号]-[名称].md`

**用途**：记录任务完整详情，包括背景、决策、技术方案

**特点**：
- 每个任务一个独立文件
- 记录完整上下文和决策过程
- 可长期归档，供后续追溯

**文件编号规则**：
- 按创建顺序编号：`001-`、`002-`、`003-`...
- 编号后跟任务名称（kebab-case 或中文）

**文件结构**：
```markdown
# [任务编号] 任务名称

## 任务信息
- **优先级**: P0/P1/P2/P3
- **负责人**: @角色

- **状态**: 待开始/进行中/审查中/已完成

## 背景与目标
[任务背景说明和目标描述]

## 相关文件

## 变更记录
| 日期 | 变更内容 | 变更人 |
|------|----------|--------|
| | | |

| | | | |
```

## 任务状态流转

```
待开始 (pending)
    ↓ (负责人确认接收)
进行中 (in_progress)
    ↓ (提交成果物)
审查中 (review)
    ↓ (审查通过/不通过)
已完成 (completed)
    ↓ (归档)
已关闭 (closed)
```

**状态定义**：

| 状态 | 说明 | 流转条件 |
|------|------|----------|
| pending | 任务已创建，等待执行 | 负责人确认接收 |
| in_progress | 任务正在执行 | 提交成果物 |
| blocked | 任务被阻塞 | 阻塞解除 → in_progress |
| review | 等待审查验收 | 审查通过 → completed |
| completed | 任务完成并验收 | 归档 → closed |
| closed | 任务取消或废弃 | - |

## 任务优先级定义

| 优先级 | 说明 | 响应时间 | 示例 |
|--------|------|----------|------|
| P0 | 紧急且重要 | 立即处理 | 线上 Bug、核心功能开发 |
| P1 | 重要不紧急 | 24 小时内 | 关键功能、重要文档 |
| P2 | 紧急不重要 | 48 小时内 | 一般优化、文档更新 |
| P3 | 不紧急不重要 | 待排期 | 技术债务、体验优化 |

## 会话流程集成

### 会话开始时

1. **读取** `.claude/TASK.md`
2. **确认** 任务状态是否准确
3. **如有变化** 先更新 TASK.md
4. **开始工作**

### 会话中

- **任务完成时** 立即标记 `[x]`
- **遇到阻塞时** 更新 `blocked` 状态
- **重大决策时** 记录到 TASK.md 和详情文档
- **任务动态调整时** 遵循 [任务动态调整流程](./TASK-动态调整示例.md)

### 会话结束时

1. **总结** 本次完成工作
2. **更新** `.claude/TASK.md` 进度
3. **记录** 下一步行动
4. **如需归档** 更新任务详情文档

## 任务动态调整（新增）

> 适用于 TASK.md 中有任务时，临时添加/删除/调整任务的场景。

### 添加任务

| 场景 | 操作 | 标记 |
|------|------|------|
| 添加待开始任务 | 添加到"待开始"列表，记录添加时间 | `← 新增 2026-03-30` |
| 插入紧急任务 | 当前任务标记"暂停"，新任务标记"插队" | `🔥 插队`、`⏸️ 已暂停` |

### 删除任务

| 场景 | 操作 | 记录位置 |
|------|------|----------|
| 删除待开始任务 | 从列表移除，"备注"记录原因 | 备注 |
| 删除进行中任务 | 从列表移除，记录已完成部分 | 备注 |

### 任务状态调整

| 场景 | 操作 | 标记 |
|------|------|------|
| 任务暂停 | 标记"⏸️ 已暂停"，记录原因 | `⏸️ 已暂停` |
| 任务恢复 | 移除暂停标记 | `▶️ 已恢复` |
| 优先级调整 | 更新优先级标记 | `P0 ← P2` |

### 变更记录格式（可选）

```markdown
## 变更记录
| 时间 | 变更类型 | 变更内容 | 原因 |
|------|----------|----------|------|
| 2026-03-30 22:45 | 添加任务 | 新增任务 XXX | 用户临时要求 |
```

**详细规范**：详见 [任务动态调整示例](./TASK-动态调整示例.md)

## 多会话锁机制

**问题**：同时打开多个 VSCode 窗口时，可能并发写入 TASK.md

**解决方案**：

```yaml
# TASK.md 头部包含
session: be2d86c6-3b77-43f3-bfba-6fadbea86dfb  # 当前会话 ID
lock: 1711785600  # Unix 时间戳（秒）
```

**写入规则**：
1. 写入前检查 `lock` 时间戳
2. 如当前时间 - lock < 300 秒（5 分钟）→ 警告"有其他会话正在工作"
3. 写入时更新 `lock` 为当前时间戳
4. 用户确认后继续
