# Rules vs 文档效果验证实验

> 任务编号：2026-04-02-1430 | 状态：已完成 | 优先级：P1

---

## 实验目的

验证"Rules/Commands 模式"是否比"纯文档模式"更能有效指导 AI 工程师行为。

---

## 背景

### 问题描述

项目中存在大量规范文档（测试规范、前端检查清单、任务执行流程等），但 AI 工程师在执行任务时：
- 未遵循文档规范
- 测试修复时直接修改代码，未验证实际 DOM
- 文档对 AI 行为影响有限

### 核心假设

- 文档 = 知识描述（AI 自行判断是否遵循）
- Rules = 可执行步骤（系统强制加载）

---

## 实验设计

### 步骤 1：分析官方模式

获取 Anthropic Claude Code 官方项目的 `.claude` 目录结构。

**结果**：
- 官方只使用 `.claude/commands/` 目录
- 没有 CLAUDE.md 文件
- 没有 rules 目录
- dedupe.md 示例包含 `allowed-tools`、分步流程、脚本封装

**结论**：官方也不依赖"文档知识"，而是使用"可执行 Commands"。

### 步骤 2：创建测试 Rule

创建 `test-fix-rule.md`，强制执行"先验证 DOM 再修改代码"流程。

**文件位置**：
- `.claude/rules/project-specific/test-fix-rule.md`
- `.claude/commands/test-fix.md`

### 步骤 3：执行验证

使用 `/test-fix` skill 执行测试修复任务。

---

## 实验结果

### Rule 强制执行的步骤

| 步骤 | 执行情况 | 发现 |
|------|----------|------|
| Step 1: 运行失败测试 | ✅ 执行 | auth.setup 登录超时 |
| Step 2: 检查错误截图 | ✅ 执行 | 页面填写了账号但未跳转 |
| Step 3: 测试 API | ✅ 执行 | **ECONNRESET** 后端连接重置 |

### 关键洞察

**如果没有 Rule**：
- AI 会按 TASK.md 说的"选择器问题"直接修改代码
- 浪费时间在无效修复上
- 问题本质是后端服务不稳定，不是选择器

**有了 Rule 后**：
- 强制执行诊断步骤
- 发现真正问题是后端服务不稳定
- 避免了无效修复

---

## 核心结论

### 文档 vs Rules/Commands 对比

| 维度 | 文档 | Rules/Commands |
|------|------|----------------|
| 内容类型 | 知识描述 | **可执行步骤** |
| 执行方式 | AI 自行判断 | **系统强制加载** |
| 验证标准 | 无 | **每步有检查点** |
| 工具限制 | 无 | **allowed-tools** |
| 官方采用 | ❌ | ✅ Anthropic 采用 |

### 验证结论

**Rules/Commands 模式比纯文档更有效**，验证通过。
