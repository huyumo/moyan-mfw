### 技术栈选择

| 项目 | 选择 | 说明 |
|------|------|------|
| 基础组件库 | TSX + `defineComponent` | 类型完整、灵活 |
| 业务组件 | `<script setup lang="ts">` | 简洁、开发效率高 |
| Vue | 3.4+ | 项目统一版本 |
| TypeScript | 5.3+ | 项目统一版本 |
| Element Plus | 2.5+ | UI 组件库 |

### 命名规范

| 项目 | 规范 | 示例 |
|------|------|------|
| 组件名前缀 | PascalCase + Mfw | `MfwFormCard` |
| 组件目录 | camel-case + 连字符 | `mfw-form-card/` |
| 主组件文件 | PascalCase | `Index.vue` |
| 子组件文件 | kebab-case | `column-control.vue` |
| composable | camel-case + use | `use-draft-dox.ts` |
| 类型定义 | camel-case + types | `types.ts` |

## 关键决策

### 决策 1：组件命名使用 `Mfw` 前缀
- **原因**：避免与 Element Plus 和其他库的组件名冲突，增强框架识别度
- **影响**：所有组件需要重命名，如 `FormCard` → `MfwFormCard`

### 决策 2：基础组件采用 TSX + `defineComponent`
- **原因**：
  - TSX 提供更好的类型推断
  - `defineComponent` 支持完整的组件选项
  - 适合复杂组件逻辑
- **影响**：需要编写 render 函数，代码量增加

### 决策 3：类型定义完整，避免使用 `any`
- **原因**：
  - 提高代码可维护性
  - 更好的 IDE 支持
  - 减少运行时错误
- **影响**：开发成本增加，但长期收益大

### 决策 4：组件分类存放
- **原因**：
  - 清晰的目录结构
  - 便于查找和维护
  - 符合框架规范
- **影响**：需要重构现有组件目录
