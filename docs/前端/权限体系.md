# 权限体系

## 概述

Moyan MFW 采用 **RBAC + 位运算** 的混合权限模型，结合角色权限控制与位运算细粒度控制，实现灵活且高性能的权限管理。

## 权限模型

### 核心概念

```
用户(User) ──N:N──→ 用户角色(UserRole) ──N:N──→ 角色(Role)
                                                      │
                                                 角色权限(RolePermission)
                                                      │
                                                      ↓
                                                  权限(Permission)
                                                      ↑
                                                 权限池(AppTypePermission)
                                                      │
                                                      ↓
                                                  应用类型(AppType)
```

### 角色层级

| 角色 | 绑定范围 | 说明 |
|------|---------|------|
| 全局角色 | 无 appId、无 appTypeId | 所有应用可用 |
| 应用类型角色 | 有 appTypeId | 该类型下的所有应用可用 |
| 应用实例角色 | 有 appId | 仅该应用实例可用 |

### 特殊角色

| 标记 | 说明 |
|------|------|
| `isBuiltin = 1` | 内置角色，不允许删除 |
| `isOwner = 1` | 拥有者角色，拥有全部权限 |
| `isDeveloper = 1`（用户级别） | 开发者，跳过所有权限检查 |

## 位运算权限

### 权限位定义

| 名称 | 位值 | 二进制 |
|------|------|--------|
| 添加 | 1n | 000000001 |
| 编辑 | 2n | 000000010 |
| 删除 | 4n | 000000100 |
| 导出 | 8n | 000001000 |
| 导入 | 16n | 000010000 |
| 审批 | 32n | 000100000 |
| 拒绝 | 64n | 001000000 |
| 发布 | 128n | 010000000 |
| 归档 | 256n | 100000000 |

### 运算示例

```typescript
// 组合权限：添加 + 编辑 + 删除 = 1 | 2 | 4 = 7
buildPerValue(['添加', '编辑', '删除'])  // 7n

// 检查权限
hasPermission('7', '添加')   // true  (7 & 1 = 1)
hasPermission('7', '导出')   // false (7 & 8 = 0)

// 解析权限
parsePerValue('7')  // ['添加', '编辑', '删除']
```

### 后端实现

1. 控制器使用 `@RequirePermission` 装饰器声明所需权限编码和权限值
2. `PermissionGuard` 拦截请求，查询当前用户角色对应的权限记录
3. 对 `sys_role_permissions` 表中的 `permissionValue` 执行位运算与检查
4. 支持多个装饰器 OR 逻辑：满足任一权限即通过

```typescript
// 声明需要用户管理的添加权限
@RequirePermission({ permCode: 'pc_root:sys:user', permissionValue: 1n })
createUser() {}

// 声明需要权限管理或PC权限管理（OR逻辑）
@RequirePermission({ permCode: 'pc_root:sys:permission-pc' })
@RequirePermission({ permCode: 'pc_root:sys:permission' })
getPermissions() {}
```

### 前端实现

1. 登录后获取权限菜单（`ApiAuthGetUserPermissions`）
2. 权限菜单同步到侧边栏，显示可访问的页面
3. 路由守卫根据权限菜单判断路由是否可访问
4. `v-permission` 指令控制按钮级权限
5. `usePermission` 钩子在代码中检查权限
6. `definePageConfig` + `permissions` 配置页面级权限

```vue
<!-- 按钮级权限控制 -->
<el-button v-permission="'添加'" type="primary">新增</el-button>
<el-button v-permission="'编辑'" type="warning">编辑</el-button>
<el-button v-permission="'删除'" type="danger">删除</el-button>
```

## 权限池机制

权限池定义每种应用类型可用的权限范围：

1. `AppTypePermission` 记录应用类型与权限的关联
2. 角色分配权限时，只能从权限池中选择
3. 权限池中的 `permissionValue` 是 `Permission.permissionValue` 的子集

### 权限池管理

- 查看权限池：`GET /api/app-types/:appTypeId/permission-pool`
- 更新权限池：`PUT /api/app-types/:appTypeId/permission-pool`

## 权限树

权限以树形结构组织，支持以下节点类型：

| 节点类型 | 说明 |
|---------|------|
| MENU | 菜单节点 |
| PAGE | 页面节点 |
| TAG | 标签节点 |

权限编码（permCode）按树结构自动组装，例如：
- `pc_root` → PC端根权限
- `pc_root:sys` → 系统管理
- `pc_root:sys:user` → 用户管理

### 前端路由同步

`POST /api/permissions/sync` 可将前端路由自动同步为 PC 权限节点，保持前后端权限定义一致。
