# 莫言权限管理系统产品概述

> 版本：1.0.0
> 最后更新：2026-03-28

---

## 1. 产品定位

莫言权限管理系统（Moyan MFW）是一款面向多租户应用的**统一权限管理平台**，采用创新的**应用类型中心模式**，为企业提供细粒度、可配置的权限控制解决方案。

### 核心价值

- **统一权限管理**：一套系统支持多个应用的权限控制
- **细粒度授权**：支持页面级、按钮级、数据级权限控制
- **灵活配置**：可视化权限池配置，无需编码即可调整权限
- **安全合规**：完整的审计日志，满足企业合规要求

## 3. 核心功能

### 3.1 应用类型管理

**功能描述**: 定义应用类型及其权限池，作为权限管理的核心节点。

**核心能力**:
- 创建/编辑/删除应用类型
- 配置应用类型权限池（可选权限范围）
- 管理内置角色（应用类型级别的全局角色）
- 配置拥有者角色（每个应用类型必须有一个）

**适用场景**:
- 企业有多个同类型应用（如多个 OA 系统、多个 CRM 系统）
- 需要统一权限模板，避免重复配置

### 3.2 权限管理

**功能描述**: 定义权限树形结构，支持位运算权限控制。

**核心能力**:
- 权限树形结构（MENU → PAGE/TAG）
- 位运算权限值（permissionValue）：ADD=1, EDIT=2, DELETE=4, ...
- 路由同步功能（自动同步前端路由到权限树）
- 权限比对功能（检测路由与权限树差异）

**权限类型**:
- **PC 权限**：页面级操作权限（新增、编辑、删除等）
- **普通权限**：标签级操作权限（查看、导出、审批等）

### 3.3 角色管理

**功能描述**: 管理角色及其权限分配，支持两种角色类型。

**核心能力**:
- 内置角色（isBuiltin=1）：应用类型级别的全局角色
- 应用级角色（isBuiltin=0）：特定应用实例的角色
- 拥有者角色（isOwner=1）：每个应用类型必须有一个，不可删除
- 权限分配（从权限池中选择）

**角色权限约束**:
- 角色权限必须是权限池的子集
- 验证公式：`(roleValue & poolValue) === roleValue`

### 3.4 用户权限

**功能描述**: 管理用户与角色、应用的绑定关系。

**核心能力**:
- 用户直接绑定角色
- 通过应用绑定角色
- 用户最终权限 = 所有关联角色权限的位运算 OR
- 权限计算层缓存（Redis 30 分钟过期）

**权限继承规则**:
```
用户最终权限 = role1Value | role2Value | role3Value | ...
```

### 3.5 审计日志

**功能描述**: 记录系统所有关键操作，支持安全审计和合规要求。

**核心能力**:
- 8 大模块：认证、用户、角色、权限、应用、应用类型、成员、系统
- 30+ 事件类型：登录、创建、更新、删除、分配等
- 数据快照：记录变更前后的数据
- 存储策略：按月分区、数据保留策略

## 4. 系统架构

### 4.1 五层架构设计

```
┌─────────────────────────────────────────────────────────────┐
│ 第一层：用户层 (User Layer)                                 │
│ - 用户基本信息、用户 - 角色绑定、用户 -应用绑定              │
└─────────────────────────────────────────────────────────────┘
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 第二层：应用实例选择层 (Application Instance Layer)         │
│ - 用户选择要访问的具体应用实例                              │
└─────────────────────────────────────────────────────────────┘
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 第三层：权限计算层 (Permission Calculation Layer)           │
│ - 收集用户所有关联角色、计算权限并集                        │
└─────────────────────────────────────────────────────────────┘
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 第四层：权限池层 (Permission Pool Layer)                    │
│ - 定义应用类型可用的权限范围、约束角色权限选择              │
└─────────────────────────────────────────────────────────────┘
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 第五层：权限定义层 (Permission Definition Layer)            │
│ - 定义所有可用的权限节点和操作、权限树形结构                │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 权限隔离机制

不同应用类型的权限池相互隔离：

```
应用类型 A (OA 系统)         应用类型 B (CRM 系统)
┌─────────────────┐         ┌─────────────────┐
│ 权限池 A         │         │ 权限池 B         │
│ [P1,P2,P3]      │         │ [P4,P5,P6]      │
│ 内置角色 A1,A2   │         │ 内置角色 B1,B2   │
│ 应用级角色 A-01  │         │ 应用级角色 B-01  │
└─────────────────┘         └─────────────────┘
```

## 5. 用户场景

### 场景 1：新员工入职

1. HR 在系统中创建用户账号
2. 主管给用户分配角色（如"审批专员"）
3. 用户登录后自动获得角色权限
4. 用户可访问的应用和菜单根据权限自动显示

### 场景 2：权限调整

1. 管理员打开权限池配置页面
2. 勾选/取消勾选权限节点
3. 设置 permissionValue（位运算权限值）
4. 保存后，相关角色的权限自动更新

### 场景 3：多应用管理

1. 用户登录后，选择要访问的应用实例
2. 系统加载用户在该应用下的权限树
3. 前端根据权限渲染菜单和按钮
4. 后端在接口调用时验证权限

### 场景 4：路由同步

1. 前端新增路由配置
2. 管理员点击"同步路由"按钮
3. 系统自动检测差异（新增、删除、变更）
4. 确认同步后，权限树自动更新

## 6. 技术特点

### 6.1 位运算权限设计

- **高效存储**：使用 bigint 类型，最多支持 64 种操作权限
- **快速计算**：位运算 OR 合并权限，AND 验证权限
- **易于扩展**：新增权限位无需修改数据库结构

**权限位定义**:
| 权限位 | 位运算值 | 说明 |
|--------|---------|------|
| ADD | 1n | 新增 |
| EDIT | 2n | 编辑 |
| DELETE | 4n | 删除 |
| EXPORT | 8n | 导出 |
| IMPORT | 16n | 导入 |
| VIEW | 32n | 查看 |
| APPROVE | 64n | 审批 |
| REJECT | 128n | 拒绝 |

### 6.2 懒清理同步策略

- **路由同步**：仅更新 sys_permission 表，不级联删除关联数据
- **孤立数据清理**：在读取时检查并清理无效数据
- **避免级联风险**：防止误删导致的权限丢失

### 6.3 前后端权限验证

| 层级 | 职责 | 实现方式 |
|------|------|----------|
| 前端 | 控制菜单/按钮显示 | 位运算检查 permissionValue |
| 后端 | 控制接口访问 | 注解式权限验证（@RequirePermission） |
| 数据库 | 数据隔离 | 行级权限过滤（可选） |

## 7. 快速开始

### 7.1 系统初始化

1. 部署系统，执行数据库初始化脚本
2. 创建超级管理员账号（admin）
3. 创建应用类型（如 OA 系统、CRM 系统）
4. 配置应用类型权限池
5. 创建内置角色（拥有者、管理员、普通用户）

### 7.2 配置第一个应用

1. 创建应用类型（如"商城系统"）
2. 配置权限池（勾选可选权限）
3. 创建内置角色（拥有者、管理员、运营、客服）
4. 创建应用实例（如"商城 - 北京"、"商城 - 上海"）
5. 分配成员和角色

### 7.3 集成到现有应用

1. 在后端添加权限验证注解（`@RequirePermission`）
2. 在前端添加权限检查函数（`hasPermission`）
3. 配置路由与权限树的同步
4. 测试权限控制逻辑
