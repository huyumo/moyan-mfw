# 术语表 (Glossary)

> 版本：1.0.0 | 最后更新：2026-03-27

本文档收集系统中的专业术语，提供中英文对照和使用场景说明。

## A

### App Type (应用类型)

| 项目 | 说明 |
|------|------|
| **英文** | Application Type |
| **缩写** | AppType |
| **对应表** | `sys_app_type` |
| **使用场景** | 定义一类应用的模板，如 OA 系统、CRM 系统、项目管理系统的类型定义 |

**说明**: 应用类型是系统的核心概念，用于定义一类应用的权限池、内置角色等配置。每个应用实例必须归属于一个应用类型。

**相关术语**: [应用实例](#app-应用实例)、[权限池](#permission-pool-权限池)

---

### App (应用实例)

| 项目 | 说明 |
|------|------|
| **英文** | Application Instance |
| **对应表** | `sys_app` |
| **使用场景** | 应用类型的实际运行实例，如"某某公司 OA 系统"是 OA 应用类型的一个实例 |

**说明**: 应用实例是用户实际访问和使用的系统实例。一个应用类型可以有多个应用实例。

**相关术语**: [应用类型](#app-type-应用类型)、[拥有者](#owner-拥有者)

---

### Audit Log (审计日志)

| 项目 | 说明 |
|------|------|
| **英文** | Audit Log |
| **对应表** | `sys_audit_log` |
| **使用场景** | 记录系统中的关键操作，用于安全审计和问题追溯 |

**说明**: 审计日志记录用户的关键操作，包括登录、权限变更、数据修改等，支持系统审计和合规性要求。

**相关文档**: [审计日志设计](../06-API 接口/审计日志设计.md)

## B

### Built-in Role (内置角色)

| 项目 | 说明 |
|------|------|
| **英文** | Built-in Role |
| **标识字段** | `sys_role.isBuiltin = 1` |
| **使用场景** | 应用类型级别的全局角色，如"OA 管理员"、"OA 普通用户" |

**说明**: 内置角色绑定到应用类型（`sys_role.appTypeId`），在该应用类型的所有实例中有效。与[应用级角色](#application-level-role-应用级角色)相对。

**相关术语**: [角色](#role-角色)、[应用级角色](#application-level-role-应用级角色)

## I

### isOwner (拥有者标识)

| 项目 | 说明 |
|------|------|
| **类型** | Boolean |
| **对应表** | `sys_role.isOwner` |
| **使用场景** | 标识一个角色是否为拥有者角色 |

**说明**: 每个应用类型必须有一个且仅有一个拥有者角色（`isOwner=1`）。拥有者角色通常是内置角色，拥有该应用类型的最高权限。

**相关术语**: [拥有者](#owner-拥有者)

## M

### Member (成员)

| 项目 | 说明 |
|------|------|
| **英文** | Member |
| **对应表** | `sys_user_app` |
| **使用场景** | 被添加到应用实例中的普通用户 |

**说明**: 成员是应用实例的普通参与者，通过角色分配获得权限。与[拥有者](#owner-拥有者)相对。

**相关术语**: [拥有者](#owner-拥有者)、[角色](#role-角色)

## O

### Owner (拥有者)

| 项目 | 说明 |
|------|------|
| **英文** | Owner |
| **对应表** | `sys_app.ownerId` |
| **使用场景** | 应用实例的负责人，每个应用实例只能有一个拥有者 |

**说明**: 拥有者是应用实例的最高负责人，自动绑定拥有者角色，拥有该应用实例的完整权限。拥有者变更时，原拥有者失去访问权限（除非同时是成员）。

**相关术语**: [成员](#member-成员)、[拥有者角色](#owner-role-拥有者角色)

---

### Owner Role (拥有者角色)

| 项目 | 说明 |
|------|------|
| **英文** | Owner Role |
| **标识字段** | `sys_role.isOwner = 1` |
| **使用场景** | 标识一个角色为拥有者角色 |

**说明**: 拥有者角色通过 `isOwner=1` 标识，每个应用类型必须有一个拥有者角色。拥有者账号自动绑定该角色，通过角色权限获得所有权限。

**相关术语**: [拥有者](#owner-拥有者)、[isOwner](#isowner-拥有者标识)

## P

### Perm (全局权限位枚举)

| 项目 | 说明 |
|------|------|
| **全称** | Permission Bit |
| **类型** | bigint (位运算) |
| **对应表** | `sys_permission.permissionValue`、`sys_app_type_permission.permissionValue`、`sys_role_permission.permissionValue` |
| **使用场景** | 定义页面内的操作权限，如"新增"、"编辑"、"删除"按钮权限 |

**说明**: Perm 是权限系统的最小粒度，使用位运算存储。它在三层结构中传递：
1. **权限定义层**: `sys_permission.permissionValue` 定义 PAGE 节点的所有可用操作（位运算值）
2. **权限池层**: `sys_app_type_permission.permissionValue` 配置应用类型可选的操作子集（位运算值）
3. **角色权限层**: `sys_role_permission.permissionValue` 分配角色实际拥有的操作（位运算值）

**全局权限位定义**:
```typescript
enum Perm {
  ADD     = 1n,    // 00001
  EDIT    = 2n,    // 00010
  DELETE  = 4n,    // 00100
  EXPORT  = 8n,    // 01000
  IMPORT  = 16n,   // 10000
  VIEW    = 32n,   // 100000
  APPROVE = 64n,   // 1000000
  REJECT  = 128n,  // 10000000
  PUBLISH = 256n,  // 100000000
  ARCHIVE = 512n,  // 1000000000
}
```

**位运算示例**:
```typescript
// ADD | EDIT = 3n (00011)
const addEdit = Perm.ADD | Perm.EDIT

// 验证是否有 ADD 权限
if ((userValue & Perm.ADD) !== 0n) { /* 有权限 */ }
```

**相关术语**: [权限](#permission-权限)、[权限池](#permission-pool-权限池)、[permissionValue](#permissionvalue-位运算权限值)

---

### permissionValue (位运算权限值)

| 项目 | 说明 |
|------|------|
| **类型** | bigint |
| **对应表** | `sys_permission.permissionValue`、`sys_app_type_permission.permissionValue`、`sys_role_permission.permissionValue` |
| **说明** | 使用位运算存储的权限值，替代原有的 pcAction JSON 数组 |

**位运算公式**:
| 操作 | 公式 | 说明 |
|------|------|------|
| 合并权限 | `a \| b` | 取并集 |
| 验证权限 | `(value & required) !== 0n` | 是否有指定权限 |
| 子集检查 | `(child & parent) === child` | child 是否是 parent 的子集 |

**相关术语**: [Perm](#perm-全局权限位枚举)、[权限](#permission-权限)

### Permission (权限)

| 项目 | 说明 |
|------|------|
| **英文** | Permission |
| **对应表** | `sys_permission` |
| **使用场景** | 控制用户可访问的菜单、页面和操作 |

**说明**: 权限是系统访问控制的基本单位，具有树形结构（MENU → PAGE）。权限通过 [PermissionType](#permissiontype-权限类型) 和 [NodeType](#nodetype-节点类型) 双层设计。
PAGE 节点的操作权限使用 [permissionValue](#permissionvalue-位运算权限值) 位运算存储。

**相关术语**: [permissionValue](#permissionvalue-位运算权限值)、[PermissionType](#permissiontype-权限类型)、[NodeType](#nodetype-节点类型)

---

### Permission Pool (权限池)

| 项目 | 说明 |
|------|------|
| **英文** | Permission Pool |
| **对应表** | `sys_app_type_permission` |
| **使用场景** | 定义应用类型可用的权限范围，约束该应用类型下所有角色的权限选择 |

**说明**: 权限池是应用类型的权限配置中心，内置角色和应用级角色的权限都必须从权限池中选择。不同应用类型的权限池相互隔离。

**相关术语**: [应用类型](#app-type-应用类型)、[权限](#permission-权限)

---

### PermissionType (权限类型)

| 项目 | 说明 |
|------|------|
| **类型** | Enum |
| **取值** | `PC` / `NORMAL` |
| **使用场景** | 区分权限是否包含 permissionValue 操作控制 |

**说明**:
- `PC`: 包含 permissionValue 操作控制的权限类型，通常用于需要细粒度控制的菜单和页面
- `NORMAL`: 普通权限类型，只有有无的区别，不包含操作控制

**相关术语**: [NodeType](#nodetype-节点类型)、[permissionValue](#permissionvalue-位运算权限值)

---

### NodeType (节点类型)

| 项目 | 说明 |
|------|------|
| **类型** | Enum |
| **取值** | `MENU` / `PAGE` / `TAG` |
| **使用场景** | 定义权限节点在树形结构中的类型 |

**说明**:
- `MENU`: 菜单目录节点，可包含子节点
- `PAGE`: 页面节点，可包含 permissionValue 操作权限
- `TAG`: 标签节点，用于普通权限分类

**组合规则**:
| PermissionType | NodeType | 说明 |
|----------------|----------|------|
| PC | MENU | PC 菜单目录 |
| PC | PAGE | PC 页面权限（父节点必须是 MENU） |
| NORMAL | MENU | 普通权限目录 |
| NORMAL | TAG | 普通权限标签 |

**相关术语**: [PermissionType](#permissiontype-权限类型)、[权限树](#permission-tree-权限树)

---

### Permission Tree (权限树)

| 项目 | 说明 |
|------|------|
| **英文** | Permission Tree |
| **使用场景** | 权限的树形组织结构 |

**说明**: 权限以树形结构组织，从根节点开始，MENU 节点可包含子 MENU 或 PAGE 节点，PAGE 节点包含 [permissionValue](#permissionvalue-位运算权限值) 位运算权限值。

**示例**:
```
ROOT (MENU)
├── system (MENU)
│   ├── user-list (PAGE)
│   │   └── permissionValue: 7n (ADD|EDIT|DELETE)
│   └── role-list (PAGE)
│       └── permissionValue: 5n (ADD|DELETE)
└── business (MENU)
    └── order-list (PAGE)
        └── permissionValue: 192n (APPROVE|REJECT)
```

**相关术语**: [权限](#permission-权限)、[permissionValue](#permissionvalue-位运算权限值)

### sys_* Tables (系统表)

| 表名 | 说明 |
|------|------|
| `sys_user` | 用户基本信息 |
| `sys_app_type` | 应用类型定义 |
| `sys_app` | 应用实例 |
| `sys_user_app` | 用户 - 应用绑定关系 |

## U

### User Role (用户角色关联)

| 项目 | 说明 |
|------|------|
| **英文** | User Role Association |
| **对应表** | `sys_user_role` |
| **使用场景** | 记录用户绑定的角色 |

**说明**: 用户通过该表与角色建立关联，从而获得角色的权限。用户可以绑定多个角色，最终权限为所有角色权限的并集。

**相关术语**: [角色](#role-角色)、[权限并集](#union-permission-权限并集)

---

### User App (用户应用关联)

| 项目 | 说明 |
|------|------|
| **英文** | User App Association |
| **对应表** | `sys_user_app` |
| **使用场景** | 记录用户可访问的应用实例 |

**说明**: 该表记录用户与应用实例的绑定关系，包括拥有者和成员两种身份。

**相关术语**: [拥有者](#owner-拥有者)、[成员](#member-成员)

---

### Union Permission (权限并集)

| 项目 | 说明 |
|------|------|
| **英文** | Permission Union |
| **使用场景** | 计算用户最终权限的方式 |

**说明**: 用户最终权限 = 所有关联角色权限的并集。使用位运算 OR 操作合并相同 `permissionId` 的 `permissionValue`。

**示例**:
```
角色 R1: {P1: 3n (ADD|EDIT), P2: 1n (ADD)}
角色 R2: {P2: 6n (EDIT|DELETE), P3: 32n (VIEW)}

用户最终权限:
{P1: 3n (ADD|EDIT), P2: 7n (ADD|EDIT|DELETE), P3: 32n (VIEW)}
```

**相关术语**: [角色](#role-角色)、[permissionValue](#permissionvalue-位运算权限值)
